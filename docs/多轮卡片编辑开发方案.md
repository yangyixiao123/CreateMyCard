# AI 卡片多轮编辑开发方案

## 1. 文档定位

本文档用于指导“基于上一版卡片继续修改”的开发、联调和测试，采用以下确定方案：

- 继续使用现有 `generateWidgetCard` 接口，不新增独立编辑接口。
- 为 `generateWidgetCard` 增加可选参数 `sourceArtifactUrl`。
- 不传 `sourceArtifactUrl` 表示首次生成；传入 `sourceArtifactUrl` 表示基于该 artifact 进行编辑。
- OBS URL 按当前项目约定视为永久有效，但来源 artifact 仍必须保持不可变；每次编辑生成新的 OBS URL。
- 主 Agent 不读取、不生成、不修改 DSL；微服务负责读取来源 artifact、重建 CardSpec、构造模型输入、生成新 DSL、校验并上传新 artifact。

具体系统边界、协议约束、状态码和 artifact 规则仍以 `docs/云侧方案设计.md` 为唯一权威来源。本文档是专题实施指导，不得用于绕过能力过滤、协议 profile 或完整 artifact 校验。

## 2. 背景与问题

首次生成链路中，主 Agent 调用 `generateWidgetCard` 后只持有工具返回的 `artifactUrl`，实际 DSL、CardSpec、TaskSpec 和有效能力信息位于 OBS artifact 中。用户在下一轮提出“背景改成蓝色”“去掉日历”“把上海改成北京”等修改需求时，主 Agent虽然有多轮对话上下文，但不应为了编辑而下载或理解 DSL。

需要解决的问题包括：

1. 如何定位需要修改的上一版卡片。
2. 如何继承未被用户修改的布局、动态能力、事件和素材。
3. 如何区分纯视觉修改与能力变化。
4. 如何保证 A2UI 模型仍只生成 DSL，不修改最终 CardSpec。
5. 如何保证编辑后的产物继续满足设备能力、协议版本和完整 artifact 校验要求。
6. 如何在不新增复杂会话存储和工作流引擎的前提下完成最小闭环。

## 3. 目标与非目标

### 3.1 目标

- 支持用户在同一会话中对上一版卡片连续修改。
- 支持纯视觉、静态文案、尺寸、数据能力、能力参数、点击事件和素材调整。
- 保持首次生成接口兼容；现有调用方不传新字段时行为不变。
- 编辑失败时不影响来源 artifact。
- 每一轮编辑都返回新的完整 artifact URL，可继续作为下一轮编辑来源。
- 编辑后的 CardSpec 和 DSL 始终经过当前设备能力裁决、协议 profile 选择和完整校验。

### 3.2 非目标

- 不让主 Agent 直接读取或修改 DSL。
- 不让 A2UI 模型生成或修改 CardSpec。
- 不支持对任意外部 URL 内容进行卡片编辑。
- 不在本期引入通用版本管理平台、可视化 diff 引擎或复杂工作流引擎。
- 不覆盖或删除历史 artifact。
- 不保证像素级局部修改；模型输出仍是完整三行 DSL。

## 4. 核心设计决策

### 4.1 接口模式判定

`generateWidgetCard` 根据 `sourceArtifactUrl` 是否存在确定处理模式：

```text
sourceArtifactUrl 缺失
 -> create 模式

sourceArtifactUrl 为合法非空字符串
 -> edit 模式

sourceArtifactUrl 已出现但为空、null 或类型错误
 -> INVALID_ARGUMENTS
```

微服务不得根据自然语言猜测当前是首次生成还是编辑，也不得在 edit 模式来源 artifact 加载失败后静默降级为 create 模式。

### 4.2 URL 的定位

`sourceArtifactUrl` 只是来源 artifact 的定位参数，不进入 A2UI 模型输入。主 Agent只需从上一轮 `generateWidgetCard` 的业务结果中原样取出 URL 并回传，不需要生成、解析或缩写 URL。

虽然 OBS URL 按项目约定永久有效，仍需满足以下约束：

- 只允许项目配置的 OBS host、bucket 和对象前缀。
- 来源对象必须是本服务曾生成的完整 artifact。
- 来源对象不可被编辑流程覆盖。
- 每次编辑上传到新的对象路径并返回新的 URL。

### 4.3 编辑输出形态

模型在编辑模式下仍输出完整 DSL，而不是 JSON Patch、组件 patch 或局部文本替换。这样可以直接复用现有 DSL 校验、完整 artifact 校验、失败重试和端侧下载渲染链路。

### 4.4 CardSpec 职责

来源 CardSpec 只供微服务解析、继承和重新裁决。微服务必须先得到新一轮最终 CardSpec，再构造新 TaskSpec 和 A2UI 模型输入。旧 CardSpec 和新 CardSpec 都不得交给 A2UI 模型自由修改。

## 5. 接口契约

### 5.1 WebSocket path

```text
/genui-agent-service/api/v1/ws/tools/generateWidgetCard
```

### 5.2 新增参数

| 字段 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| `sourceArtifactUrl` | `String` | 否 | 上一版完整 artifact 的永久 OBS URL。缺失表示首次生成，非空表示二次编辑。 |

### 5.3 首次生成示例

```json
{
  "userQuery": "帮我做一个上海天气卡片",
  "size": "2x2",
  "title": "上海天气",
  "description": "今日天气速览",
  "candidateDataBindings": [
    {
      "capabilityId": "ViewWeather",
      "arguments": {
        "districtName": "上海",
        "forecastDays": 1
      },
      "writeResultTo": "/data/weather"
    }
  ],
  "candidateEventCandidates": [],
  "candidateAssetIds": ["asset.weather.rain"]
}
```

### 5.4 纯视觉编辑示例

```json
{
  "userQuery": "整体改成蓝色风格，标题更醒目",
  "sourceArtifactUrl": "https://obs.example/widget/8f20c714.json"
}
```

该请求省略尺寸、标题、说明和所有候选数组，表示全部继承来源 artifact，只修改 DSL 表达。

### 5.5 修改静态文案示例

```json
{
  "userQuery": "标题改成每日天气，说明改成出门前看一眼",
  "sourceArtifactUrl": "https://obs.example/widget/8f20c714.json",
  "title": "每日天气",
  "description": "出门前看一眼"
}
```

### 5.6 修改动态能力示例

```json
{
  "userQuery": "去掉日历，只保留上海天气",
  "sourceArtifactUrl": "https://obs.example/widget/8f20c714.json",
  "candidateDataBindings": [
    {
      "capabilityId": "ViewWeather",
      "arguments": {
        "districtName": "上海",
        "forecastDays": 1
      },
      "writeResultTo": "/data/weather"
    }
  ]
}
```

编辑模式下，只要显式传入某类候选数组，该数组就表示编辑后该类别的完整期望集合，而不是增量追加。

### 5.7 清空某类能力示例

```json
{
  "userQuery": "去掉所有动态数据，只保留一个打开天气应用的入口",
  "sourceArtifactUrl": "https://obs.example/widget/8f20c714.json",
  "candidateDataBindings": [],
  "candidateEventCandidates": [
    {
      "capabilityId": "event.open.weather",
      "action": {
        "call": "clickToDeeplink",
        "args": {
          "bundleName": "",
          "abilityName": "",
          "uri": "hww://www.huawei.com/totemweather?enterType=share&cityCode="
        }
      }
    }
  ]
}
```

### 5.8 编辑模式字段继承语义

编辑模式必须严格区分“字段省略”和“显式空值”：

| 字段 | 省略时 | 显式传入时 |
| --- | --- | --- |
| `size` | 继承来源 CardSpec 的 `suggestSize` | 使用本轮建议值并由微服务重新裁决 |
| `title` | 继承来源 CardSpec 标题 | 使用本轮建议值并规范化 |
| `description` | 继承来源 CardSpec 说明 | 使用本轮建议值并规范化 |
| `candidateDataBindings` | 从来源 CardSpec 还原并重新校验 | 替换为本轮完整候选集合；`[]` 表示清空 |
| `candidateEventCandidates` | 从来源 artifact 还原并重新校验 | 替换为本轮完整候选集合；`[]` 表示清空 |
| `candidateAssetIds` | 从来源 artifact 还原并重新校验 | 替换为本轮完整候选集合；`[]` 表示清空 |

约束：

- `userQuery` 在 edit 模式下只表达本轮修改需求，不要求主 Agent 拼接原始需求。
- 微服务可以读取来源 `taskSpec.userQuery` 作为历史上下文，但不得用历史需求覆盖本轮明确修改。
- 首次生成模式仍要求主 Agent 提供非空 `title` 和 `description`。
- 编辑模式允许省略 `title` 和 `description`，省略时继承来源 CardSpec。
- 若工具框架无法区分字段缺失与 `null`，主 Agent 不得传 `null`；服务端将 `null` 按参数非法处理。

### 5.9 输出

业务输出保持现有结构：

```json
{
  "status": "success",
  "artifactUrl": "https://obs.example/widget/9a31d825.json",
  "artifactDigest": "sha256:xxx",
  "suggestSize": "2x2",
  "message": "已按你的要求修改卡片。",
  "removedCapabilities": [],
  "errorCode": ""
}
```

要求：

- `success` 或 `degraded` 必须返回新的真实 `artifactUrl`。
- 不得把来源 URL 当作编辑成功结果返回。
- `unsupported` 或 `failed` 不输出新的卡片结果标记。
- 编辑失败时来源 artifact 保持可用。

## 6. 主 Agent / Skill 编排

### 6.1 识别编辑请求

满足以下条件时按编辑请求处理：

- 当前会话中存在上一轮成功或降级生成返回的真实 `artifactUrl`；
- 用户明确表达修改、调整、替换、删除、增加、换一种样式或继续优化上一张卡片。

如果会话中存在多张卡片且无法确定目标，主 Agent应先让用户明确要修改哪一张，不能自行选择 URL。

### 6.2 获取来源 URL

主 Agent 从上一轮工具业务 payload 中读取真实 `artifactUrl`，原样写入 `sourceArtifactUrl`。不得：

- 从普通自然语言中猜测 URL；
- 自行拼接 OBS 地址；
- 使用示例 URL；
- 把 `genWidgetResult` 代码块整体作为 URL；
- 下载来源 artifact 后再调用工具。

### 6.3 编辑类型与工具调用

#### 纯视觉或布局修改

示例：“背景换成蓝色”“信息更紧凑”“突出温度”。

- 只传 `userQuery` 和 `sourceArtifactUrl`。
- 未明确要求变化的尺寸、文案、能力、事件和素材全部省略。
- 不需要重新调用数据能力 schema 接口。

#### 标题、说明或尺寸修改

- 传 `userQuery`、`sourceArtifactUrl` 和明确变化的字段。
- 未变化字段省略。
- 尺寸变化后由微服务重新选择布局并完成完整校验。

#### 数据能力或能力参数修改

示例：“去掉日历”“增加电量”“把上海换成北京”。

- 主 Agent重新获取必要的能力概述和新增或变化能力的完整 schema。
- `candidateDataBindings` 必须传编辑后的完整候选集合。
- 明确删除全部动态能力时传 `candidateDataBindings: []`。
- 参数仍只能来自能力 schema 或用户明确输入。

#### 点击事件或素材修改

- 对应候选数组必须传编辑后的完整集合。
- 删除全部时显式传空数组。
- 未变化类别省略并由微服务继承。

### 6.4 用户回复

- `success`：使用微服务 `message`，输出指向新 URL 的 `genWidgetResult`。
- `degraded`：说明无法保留或新增的内容，输出指向新 URL 的 `genWidgetResult`。
- `unsupported`：说明当前修改无法形成有价值的卡片，不输出结果标记。
- `failed`：说明本次修改未完成，原卡片不受影响，不输出结果标记。

普通用户回复不得暴露来源 artifact、内部能力 ID、TaskSpec、校验错误明细或 OBS 实现细节。

## 7. 微服务处理流程

### 7.1 总体流程

```text
接收 generateWidgetCard 请求
 -> 根据 sourceArtifactUrl 判断 create/edit 模式
 -> edit 模式校验并下载来源 artifact
 -> 解析来源 genui/CardSpec/TaskSpec/有效能力/meta
 -> 按字段省略/替换语义构造编辑后候选计划
 -> 查询 IDS，重新裁决当前设备能力
 -> 选择当前适用的协议 profile
 -> 生成新 CardSpec
 -> 生成新 TaskSpec
 -> 构造编辑模型输入
 -> A2UI 模型输出完整三行 DSL
 -> 校验完整新 artifact
 -> 失败时按现有规则最多重试 1 次
 -> 上传新的 OBS 对象
 -> 返回新 artifactUrl 和结构化状态
```

### 7.2 来源 URL 校验

下载前至少校验：

1. 字段类型为字符串且非空。
2. URL 使用 `https`。
3. host、bucket、对象前缀位于服务端配置白名单。
4. URL 不包含用户可控重定向目标。
5. 对象属于卡片 artifact 存储空间。
6. 当前调用用户有权编辑该对象。
7. 下载设置连接超时、读取超时和最大响应体限制。

禁止服务端根据该字段访问任意公网、内网地址或本地文件。永久有效不等于无需鉴权和来源限制。

### 7.3 来源 artifact 校验

来源 artifact 在进入编辑流程前至少满足：

- JSON 可解析。
- `schemaVersion` 受当前服务支持。
- `genui` 和 `cardSpec` 存在。
- `genui` 是合法的三行 JSONL。
- CardSpec 基础结构合法。
- 来源尺寸和协议 profile 可识别。
- 内容大小不超过配置上限。

`taskSpec`、有效能力和 meta 缺失时，只有在当前 artifact schema 明确允许且服务能够从 CardSpec 安全还原的情况下才能继续；否则返回来源产物无效，不得猜测动态能力。

### 7.4 候选继承与替换

微服务新增统一的编辑合并步骤：

```text
EditRequestNormalizer
  输入：来源 artifact + 本轮请求
  输出：编辑后的完整候选计划
```

处理规则：

1. 从来源 CardSpec 还原数据能力候选。
2. 从来源 artifact 的有效事件和 TaskSpec 还原事件候选。
3. 从来源 artifact 的有效素材和 TaskSpec 还原素材候选。
4. 对请求中省略的类别使用来源候选。
5. 对请求中出现的类别整体替换，空数组表示清空。
6. 所有继承和替换后的候选都重新进入能力过滤，不直接复用旧裁决结果。

不得把来源 artifact 的 `effectiveCapabilities` 直接复制为新 artifact 的最终有效能力。

### 7.5 当前设备能力重新裁决

每次编辑都使用工具层自动携带的当前用户和设备上下文重新查询 IDS，并按现有固定顺序过滤：

```text
能力是否注册
 -> ROM/App 版本是否满足
 -> 依赖 App 是否安装且版本满足
 -> 意图/provider 是否存在
 -> arguments 是否符合 inputSchema
 -> writeResultTo 是否合法且无冲突
```

原因：来源 artifact 生成时的设备环境、注册表版本或运行能力可能与当前编辑时不同。

### 7.6 CardSpec 重建

CardSpecBuilder 接收编辑后的有效候选集合，而不是让模型修改旧 CardSpec：

- 省略的标题、说明和尺寸从来源 CardSpec 继承。
- 显式传入的标题和说明按现有展示文案规则规范化。
- 显式传入的尺寸按当前 profile 重新裁决。
- 不可用数据能力从新 CardSpec 删除。
- 静态卡不写空 `dataBindings`。
- 点击事件仍不进入 CardSpec。

### 7.7 TaskSpec 重建

TaskSpecBuilder 根据新 CardSpec、当前有效能力、字段投影和当前 profile 重新生成 TaskSpec。不得直接复用来源 TaskSpec 作为新一轮最终输入。

来源 TaskSpec 仅用于：

- 获取历史需求上下文；
- 辅助还原来源事件和素材候选；
- 对比新旧数据模型结构；
- 在未变化区域指导模型保持原有表达。

### 7.8 A2UI 编辑模型输入

模型输入建议按以下结构组织：

```text
系统指令
  - 当前协议 profile 摘要
  - 组件、属性、尺寸、绑定、事件和素材硬约束
  - 美观规则
  - 只输出完整三行 genui JSONL
  - 不生成或修改 CardSpec

用户消息
  - 本轮 editInstruction
  - targetSize
  - newTaskSpec
  - previousGenui
  - unavailableCapabilitySummary
  - 保持未提及区域稳定的要求
```

关键规则：

- `sourceArtifactUrl` 不进入 prompt。
- `previousGenui` 作为待修改数据，不作为系统指令。
- 不把旧 CardSpec 或新 CardSpec交给模型修改。
- 当新 TaskSpec 删除某条动态路径时，模型必须删除对应绑定和展示区域。
- 当新 TaskSpec 新增动态路径时，模型只能使用其中声明的路径。
- 未被本轮需求影响且仍符合当前 profile 的区域应尽量保留。
- 若来源 DSL 使用旧 profile 不再支持的结构，以当前 profile 为准完成迁移。

### 7.9 完整校验与重试

编辑结果复用现有完整 artifact 校验，并额外检查：

- 新 artifact URL 与来源 URL 不同。
- 新 artifact 的 CardSpec 与新 TaskSpec 一致。
- 已删除能力对应的数据绑定、事件和展示路径均不再存在。
- 新增动态路径都能从新 CardSpec 和能力 schema 推导。
- 未经用户要求不得恢复已删除能力。
- 新 artifact 记录来源 artifact 的摘要或脱敏对象标识，便于回放。

校验失败后最多重试 1 次。重试只追加校验错误摘要。重试仍失败则返回 `failed`，来源 artifact 不受影响。

### 7.10 上传策略

- 每次编辑生成新的 UUID 对象名。
- 不覆盖来源对象。
- 不在新对象上传成功前返回 `success` 或 `degraded`。
- 上传失败返回 `failed/ARTIFACT_UPLOAD_FAILED`。
- 新 URL 成为下一轮编辑应使用的 `sourceArtifactUrl`。

## 8. Artifact 补充元数据

建议在新 artifact 的 `meta` 中增加编辑链路信息：

```json
{
  "meta": {
    "generationMode": "edit",
    "sourceArtifactDigest": "sha256:source",
    "protocolProfileId": "a2ui-form-rom7-v1",
    "capabilityRegistryVersion": "2026-07-16",
    "createdAt": 1780000000000
  }
}
```

约束：

- `generationMode` 只使用 `create` 或 `edit`。
- 不把完整 `sourceArtifactUrl` 写入 artifact；使用摘要或脱敏对象标识，避免扩大链接传播范围。
- 来源摘要仅用于链路追踪，不替代新 artifact 自身摘要。
- 若当前 schema 不允许扩展 meta，应先升级 artifact schema 并完成兼容评审。

## 9. 状态与错误处理

### 9.1 状态语义

| 状态 | 编辑场景语义 |
| --- | --- |
| `success` | 本轮修改完整满足并生成新 artifact。 |
| `degraded` | 部分修改或来源能力当前不可用，已生成仍有价值的新 artifact。 |
| `unsupported` | 修改目标受能力或协议限制，无法形成有价值的新卡片。 |
| `failed` | 来源加载、模型、校验、上传或其它工程步骤失败。 |

### 9.2 新增错误码建议

```text
SOURCE_ARTIFACT_URL_INVALID
SOURCE_ARTIFACT_FORBIDDEN
SOURCE_ARTIFACT_NOT_FOUND
SOURCE_ARTIFACT_DOWNLOAD_FAILED
SOURCE_ARTIFACT_SCHEMA_UNSUPPORTED
SOURCE_ARTIFACT_INVALID
```

继续复用：

```text
UNKNOWN_CAPABILITY
INVALID_ARGUMENTS
NO_EFFECTIVE_CAPABILITY
A2UI_GENERATION_FAILED
VALIDATION_FAILED
ARTIFACT_UPLOAD_FAILED
TIMEOUT
```

处理原则：

- 来源 URL 非法、越权、不可下载或内容不合法时返回 `failed`。
- 来源 schema 明确不受支持且无法安全迁移时返回 `failed`。
- 用户要求的能力不存在且静态卡仍有价值时可返回 `degraded`。
- 用户明确要求的核心效果受协议限制且无合理替代表达时返回 `unsupported`。
- edit 模式任何失败都不得自动创建一张与来源无关的新卡片。

## 10. 模块改造建议

```text
WidgetGenerationService
├─ GenerateWidgetRequest
│  └─ sourceArtifactUrl?: string
├─ GenerationModeResolver
├─ SourceArtifactUrlValidator
├─ ArtifactRepository
│  └─ download(sourceArtifactUrl)
├─ SourceArtifactParser
├─ EditRequestNormalizer
├─ DeviceCapabilityResolver
├─ CardSpecBuilder
├─ TaskSpecBuilder
├─ PromptBuilder
│  ├─ buildCreatePrompt(...)
│  └─ buildEditPrompt(...)
├─ A2UIModelClient
├─ ArtifactValidator
├─ RetryController
└─ ArtifactUploader
```

建议不要在 Controller 中直接完成下载、继承、合并和 prompt 拼接，避免接口层承担业务编排。

### 10.1 伪代码

```text
generateWidgetCard(request, context):
    mode = sourceArtifactUrl is present ? EDIT : CREATE

    if mode == EDIT:
        validateSourceUrl(request.sourceArtifactUrl, context.uid)
        sourceArtifact = artifactRepository.download(request.sourceArtifactUrl)
        validateSourceArtifact(sourceArtifact)
        normalizedRequest = editRequestNormalizer.merge(sourceArtifact, request)
    else:
        normalizedRequest = validateCreateRequest(request)

    deviceState = idsClient.resolve(context.device, context.uid)
    effectivePlan = capabilityResolver.resolve(normalizedRequest, deviceState)
    profile = protocolRegistry.select(context.device, context.appVersion)
    cardSpec = cardSpecBuilder.build(normalizedRequest, effectivePlan, profile)
    taskSpec = taskSpecBuilder.build(normalizedRequest, cardSpec, effectivePlan)

    prompt = mode == EDIT
        ? promptBuilder.buildEditPrompt(request.userQuery, sourceArtifact.genui, taskSpec, profile)
        : promptBuilder.buildCreatePrompt(request.userQuery, taskSpec, profile)

    genui = modelClient.generate(prompt)
    artifact = artifactBuilder.build(genui, cardSpec, taskSpec, effectivePlan, mode)
    validateAndRetryAtMostOnce(artifact)
    newUrl = artifactUploader.uploadToNewObject(artifact)
    return buildBusinessResponse(newUrl, artifact)
```

## 12. 测试方案

### 12.1 接口兼容

- 不传 `sourceArtifactUrl`，首次生成行为与当前版本一致。
- 传空字符串按未传处理或参数非法处理，服务端行为必须固定；建议按未传处理仅限工具层主动剔除空值，微服务收到空字符串则返回 `INVALID_ARGUMENTS`。
- 传 `null` 返回 `INVALID_ARGUMENTS`。
- 旧版 Skill 不传新字段仍能正常生成。

### 12.2 编辑功能

- 只修改背景色。
- 调整布局密度但保持全部动态绑定。
- 修改标题和说明。
- `2x2` 改为 `2x4`。
- 删除一个数据能力，保留其它能力。
- `candidateDataBindings: []` 清空全部动态能力。
- 新增数据能力。
- 修改已有能力参数，例如上海改为北京。
- 删除点击事件。
- 替换素材。
- 连续三轮编辑，每轮使用上一轮返回的新 URL。

### 12.3 能力重新裁决

- 来源能力当前仍可用。
- 来源能力在编辑时已不可用，返回 degraded。
- 新增能力当前不可用。
- 所有核心能力均不可用但静态卡仍有价值。
- 所有核心能力均不可用且静态卡无价值。
- 来源 `writeResultTo` 在新候选合并后产生冲突。

### 12.4 来源 artifact 异常

- 非白名单 host。
- 非 HTTPS URL。
- URL 指向非 artifact JSON。
- JSON 损坏。
- 缺少 `genui`。
- 缺少 `cardSpec`。
- 不支持的 schemaVersion。
- 超过最大文件大小。
- 当前用户无权编辑。
- 下载超时或 OBS 返回异常。

### 12.5 模型和校验

- 模型返回非三行 DSL。
- 模型删除仍需保留的动态绑定。
- 模型继续引用已删除能力路径。
- 模型使用新 TaskSpec 未声明的动态路径。
- 模型使用非法组件、属性、事件或素材。
- 第一次失败、重试成功。
- 重试后仍失败，来源 artifact 不受影响。

### 12.6 OBS 与不可变性

- 编辑成功返回的新 URL 与来源 URL 不同。
- 来源对象内容未变化。
- 同一来源并发发起两次编辑，生成两个独立新 artifact。
- 新 artifact 下载摘要与返回摘要一致。
- 上传失败不返回成功状态和结果标记。

## 14. 灰度与回滚

### 14.1 灰度开关

建议增加服务端开关：

```text
widgetEditEnabled
```

灰度顺序：

1. 开发环境开放纯视觉编辑。
2. 测试环境开放文案和尺寸编辑。
3. 灰度开放数据能力删除和参数修改。
4. 最后开放新增能力、事件和素材变更。

### 14.2 回滚策略

- 关闭 `widgetEditEnabled` 后，带 `sourceArtifactUrl` 的请求返回明确的暂不支持，不得按首次生成处理。
- 首次生成链路不受开关影响。
- 已生成的历史 artifact 永久保留并可继续由端侧下载。
- 回滚不得修改或删除来源和编辑后 artifact。

## 16. 验收标准

满足以下条件才算完成：

- 首次生成接口完全兼容。
- 主 Agent只需携带上一轮真实 `artifactUrl`，不接触 DSL。
- 纯视觉编辑只传两个业务字段即可成功。
- 省略字段、替换数组和空数组语义有自动化测试覆盖。
- 每轮编辑重新执行设备能力裁决、CardSpec/TaskSpec 构建和完整 artifact 校验。
- A2UI 模型只输出 DSL，不能修改 CardSpec。
- 编辑成功返回新的永久 OBS URL，来源 URL 内容不变。
- 编辑失败不会损坏或覆盖原卡片。
- 连续多轮编辑能够始终以上一轮新 URL 为来源继续执行。
- success/degraded/unsupported/failed 状态、错误码、日志和用户话术完整闭环。
