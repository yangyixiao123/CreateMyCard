# AI 卡片项目云侧方案设计

本文档是 AI 卡片云侧方案的唯一权威来源。 当本文档与代码实际实现、其它文档内容、skill内容存在冲突时，请询问用户，以哪个为准，并给出你的建议，用户确认后刷新对应的文件内容，保持文档一致性。

核心原则是：**优先由微服务完成真实能力裁决、最终 CardSpec、A2UI 模型输入构造、DSL 生成、校验、降级和落库；主 Agent 做轻量 SOP、候选选择和用户回复组织；A2UI 模型只生成 genui DSL；端侧负责下载 artifact、渲染、确认添加和运行时刷新。若 `generateWidgetCard` 工具不可用、调用失败或返回产物不符合预期，主 Agent 可以进入兜底链路生成最终可交付结果，但不得伪造动态能力或 artifact 链接。**

主 Agent 不承诺最终一定能生成某类动态卡片，等微服务返回后再组织用户回复。

## 方案目标与范围

本项目服务于小艺 App 的 AI 卡片生成功能。用户在小艺 App 中用自然语言表达卡片需求，例如“帮我做一个通勤日常卡片，包含今天天气和今日会议”，云侧生成可在端侧预览和添加到桌面的 HarmonyOS A2UI Form 卡片。

目标链路是：

```text
用户在小艺 App 发起卡片生成请求
 -> 主 Agent 识别场景并选择候选能力
 -> 优先由微服务按用户设备实际能力裁决并生成 artifact
 -> 如生成工具不可用或产物不符合预期，主 Agent 进入兜底链路生成最终可交付结果
 -> 端侧下载 artifact，渲染 DSL 并持久化 DSL + CardSpec
 -> 卡片运行时按 CardSpec 调端侧数据能力刷新 DataModel
```

一期重点只覆盖一方应用和系统能力，例如天气、日历、系统设置、运动健康等。三方 App 数据、跨端数据和复杂页面型需求不作为默认动态能力支持，必须通过能力注册表明确声明后才能生成动态 CardSpec。本方案也暂未考虑需要授权的情况。

其中 App 中控与主 Agent 能力框架本身已具备，云侧主要需要开发的是 Skill 与微服务。

## 端到端流程

```text
Step 1: App 中控根据 App 版本和 ROM 版本决定是否加载 Skill
        ROM < 7.0: 不加载本 Skill
        ROM >= 7.0: 允许 Skill Search / Skill 加载

Step 2: 主 Agent 判断用户 query 是否匹配卡片创建场景

Step 3: 主 Agent 调微服务获取能力概述
        返回数据能力概述、不可用数据能力、事件能力概述、素材概述

Step 4: 主 Agent 根据 query 筛选候选能力
        unavailableCapabilities 存在且非空时，先排除其指示的不可用数据能力
        数据能力按需调用微服务加载完整 schema
        事件能力和素材能力可在 Step 3 直接返回候选信息

Step 5: 主 Agent 将 query、候选 dataBindings、候选事件、候选素材、size 等信息传给微服务
        工具层自动携带用户 App 版本、ROM 版本、device、uid 等上下文

Step 6: 微服务生成卡片
        6.1 查询 IDS，获取用户实际安装应用、版本、意图/provider 列表
        6.2 过滤得到用户实际可用能力集合
        6.3 根据小艺 App 版本 + ROM 版本选择 A2UI 协议 profile
        6.4 生成最终 CardSpec，构造 TaskSpec / A2UI prompt
        6.5 调 A2UI 模型生成 genui
        6.6 校验完整 artifact；失败最多重新生成 1 次
        6.7 生成 artifact 文件，上传 OBS，返回链接和结构化状态

Step 7: 主 Agent 收到微服务结果，向用户回复
        success/degraded: 返回卡片生成结果标记
        unsupported/failed: 返回用户可理解说明和替代建议
        generateWidgetCard 不可用、调用失败或产物不符合预期: 进入主 Agent 兜底生成链路，不伪造 genWidgetResult

Step 8: 端侧解析文件链接，下载 artifact，渲染 DSL 并持久化 DSL + CardSpec
        如果走主 Agent 兜底链路，则按兜底结果形态交付，不依赖云侧 artifactUrl

Step 9: 卡片安装后由端侧负责运行时数据刷新
        9.1 根据 CardSpec 的 dataBindings 和刷新计划确定待执行的数据能力
        9.2 调用端侧数据能力，取得结构符合对应能力 outputSchema 的返回值
        9.3 以 writeResultTo 作为 path、能力返回值作为 value，自动构造新的 A2UI updateDataModel 消息
        9.4 将 updateDataModel 消息注入已渲染 surface 的 A2UI 运行时
        9.5 DataModel 更新后，由已绑定该路径的组件自动重渲染
```

## 系统边界

### App 中控

职责：

- 根据 ROM/App 版本决定是否加载本 Skill。
- ROM < 7.0 时，不召回本 Skill。
- 将必要设备上下文放入会话或工具调用上下文。

不负责：

- 不做能力过滤。
- 不做 DSL 生成。
- 不生成 CardSpec。

### 主 Agent / Skill

职责：

- 识别卡片创建场景。
- 调 `getWidgetCapabilityOverview` 获取能力概述。
- `unavailableCapabilities` 存在且非空时，从能力概述中排除其指示的不可用数据能力。
- 根据用户 query 选择候选数据能力、候选事件能力和候选素材。
- 按需调 `getDataCapabilitySchemas` 获取数据能力完整 schema。
- 生成候选 `candidateDataBindings` 或候选能力计划。
- 调 `generateWidgetCard`。
- 根据微服务返回的 `status/message/artifactUrl` 组织自然语言回复。
- 当 `generateWidgetCard` 不可用、调用失败或返回产物不符合预期时，进入兜底链路生成最终可交付结果。

不负责：

- 正常链路不直接生成 `genui`。
- 正常链路不生成最终 CardSpec。
- 不判断用户设备上能力最终可用。
- 不维护完整 A2UI 协议、组件白名单、样式白名单或美观规则。
- 不向用户提前承诺“可以生成某个动态卡片”。

兜底例外：

- 仅当 `generateWidgetCard` 工具不可用、调用失败或返回产物不符合预期时，主 Agent 才允许生成最终结果。
- 兜底结果不得伪造 `artifactUrl` 或 `genWidgetResult` 标记。
- 兜底结果不得伪造动态数据能力、事件能力或素材能力；只能使用已获取的 schema、能力概述、用户明确输入和受控参考资料。
- 兜底结果应明确说明当前未走云侧 artifact 生成链路。

推荐用户过程话术：

```text
我先检查当前设备支持情况，然后为你生成可用的卡片。
```

### 微服务

职责：

- 维护能力注册表和版本兼容要求。
- 查询 IDS 获取用户实际设备能力。
- 过滤候选能力，得到实际可用能力。
- 生成最终 CardSpec。
- 根据版本选择 A2UI 协议 profile。
- 构造 TaskSpec、A2UI system prompt 和 user message。
- 调 A2UI 模型生成 DSL。
- 校验 DSL/CardSpec/artifact。
- 处理降级、拒答、失败重试。
- 上传 OBS，返回 artifact 链接、结构化状态和用户可读信息。

第一期不要引入复杂工作流引擎，普通服务编排即可。

### A2UI 模型服务

职责：

- 根据微服务构造的输入生成 `genui` DSL。

不负责：

- 不查询用户设备能力。
- 不生成或修改最终 CardSpec。
- 不判断用户是否安装 App。
- 不自行选择未过滤能力。

### 端侧

职责：

- 解析 `genWidgetResult` 标记。
- 下载 artifact。
- 渲染 DSL。
- 用户确认后持久化 DSL + CardSpec。
- 根据 CardSpec 的 `dataBindings` 和刷新计划调度运行时刷新。
- 解析每个 binding 的 `capabilityId`、`arguments` 和 `writeResultTo`，调用对应端侧数据能力。
- 保证或校验能力返回值结构符合该能力注册信息中的 `outputSchema`。
- 将能力返回值自动编译为新的 A2UI `updateDataModel` 消息并注入对应 surface；其中 `path` 取自 `writeResultTo`，`value` 为能力返回值。
- 由 A2UI `DataModel` 与组件绑定机制完成局部状态更新和界面重渲染，不要求云侧参与每次刷新。

不负责：

- 不在端侧重新生成或修改云侧下发的组件 DSL 和 CardSpec 契约。
- 不要求把运行时能力返回值上传给卡片生成云服务后再刷新。

## Skill 改造目标

当前 Skill 中很多细节需要下沉：

- 下沉到微服务：A2UI 协议、组件限制、美观规则、素材合法性、DSL 校验、重试修复、最终 CardSpec。
- 保留在 Skill：SOP 流程、能力概述筛选、候选 dataBindings 生成、工具调用、用户回复组织。
- Skill 正常链路不再直接输出 `genui/cardspec`，而是输出微服务返回的 `genWidgetResult` 链接和自然语言说明；兜底链路可输出必要最终产物，但必须说明未走云侧 artifact 生成链路。

## 主 Agent 调微服务接口

微服务以工具形式提供三个能力。主 Agent 正式调用工具时使用 `invoke(functionName:"工具名", arguments:{bundleName:"com.omega_w_0823.hmservice", ...业务参数},"skillName":"harmony-card-generation-online")`，`arguments` 包含工具 bundleName 和业务参数，不要再包 `params` 外层对象；`skillName` 必须传当前 Skill frontmatter 的 `name`，本 Skill 固定为 `harmony-card-generation-online`，不得省略或传空字符串。线上工具层会默认把用户实际设备环境信息拼接到工具入参中，示例如下：

```json
{
  "uid": "...",
  "device": {
    "deviceId": "5e64f3e9-0a80-d719-d689-3c36eca5eeb6",
    "deviceType": "ALN-AL00",
    "sysVersion": "EmotionUI_9.0.0",
    "deviceName": "phone",
    "odid": "5e64f3e9-0a80-d719-d689-3c36eca5eeb6",
    "udid": "5e64f3e9-0a80-d719-d689-3c36eca5eeb6",
    "romVersion": "ALN-AL00 6.0.0.36",
    "marketingName": "HUAWEI Mate 60 Pro",
    "ohosApiVersion": 36
  }
}
```

其中 `uid` 和 `device` 为自动拼接，无需主 Agent 生成。`DeviceContext` 至少需要 `romVersion` 和 `ohosApiVersion`。

主 Agent 每次调用 `invoke` 前必须以当前运行时 `tools` 中对应工具的 schema 为入参唯一依据：

- `functionName` 必须等于工具 schema 的工具名，`bundleName` 必须等于对应工具声明的 bundleName。
- `skillName` 必须与当前 Skill frontmatter 的 `name` 完全一致，即 `harmony-card-generation-online`。
- 除 `bundleName` 外，`arguments` 顶层只能包含该工具 schema 的 `arguments.properties` 已声明字段。
- 必须满足 schema 的 `required`、字段类型、数组元素类型和已声明的嵌套结构；无法可靠补齐必填字段时不得调用。
- schema 未声明的字段一律不传，不得根据 Skill 文案、参考资料、历史示例、内部类名或经验自行补充。
- Skill 文案、参考资料或示例与当前运行时工具 schema 不一致时，以当前运行时工具 schema 为准，并删除不一致入参。
- 数据能力候选中的能力业务参数还必须符合 `getDataCapabilitySchemas` 本轮返回的对应 `inputSchema`，不得编造字段。

主 Agent 调用顺序示例：

```text
invoke(functionName:"getWidgetCapabilityOverview", arguments:{bundleName:"com.omega_w_0823.hmservice", ...},"skillName":"harmony-card-generation-online")
invoke(functionName:"getDataCapabilitySchemas", arguments:{bundleName:"com.omega_w_0823.hmservice", ...},"skillName":"harmony-card-generation-online")
invoke(functionName:"generateWidgetCard", arguments:{bundleName:"com.omega_w_0823.hmservice", ...},"skillName":"harmony-card-generation-online")
```

### getWidgetCapabilityOverview

用途：返回主 Agent 可用于候选筛选的能力概述。

WebSocket path:

```text
/genui-agent-service/api/v1/ws/tools/getWidgetCapabilityOverview
```

输入：

```json
{}
```

输出应为结构化 JSON，不要只返回自然语言文本。可以额外提供面向 LLM 的描述字段。

```json
{
  "dataCapabilities": [
    {
      "id": "ViewWeather",
      "description": "查询当前天气、空气质量和未来预报"
    }
  ],
  "unavailableCapabilities": ["GetAppUsageDurationAndPower"],
  "eventCapabilities": [
    {
      "id": "event.open.weather",
      "call": "clickToDeeplink",
      "description": "打开天气应用"
    }
  ],
  "assetCandidates": [
    {
      "id": "asset.weather.rain",
      "description": "天气雨伞图标"
    }
  ]
}
```

说明：

- 数据能力只返回概述，不返回完整 schema。
- `unavailableCapabilities` 是可选的不可用数据能力 ID 字符串数组。字段缺失或为 `[]` 时表示没有预先标记的不可用数据能力，主 Agent 继续从全部 `dataCapabilities` 中筛选。
- `unavailableCapabilities` 非空时，主 Agent 必须先从 `dataCapabilities` 中排除这些 ID，再做候选筛选；同一 ID 同时出现时以不可用为准。不得为其中的能力调用 `getDataCapabilitySchemas`，也不得将其写入 `candidateDataBindings`。
- 字段存在但不是字符串数组时，按能力概述 payload 无法完整解析处理。
- 事件能力和素材候选体积较小，可以在该接口中返回较完整信息。
- 主 Agent 可见出参不暴露 `apiVersion`、`capabilityRegistryVersion`、`protocolProfileId` 等版本信息；这些信息只进入日志、artifact meta 和排障链路。

### getDataCapabilitySchemas

用途：针对主 Agent 已选中的数据能力渐进加载完整 schema。

WebSocket path:

```text
/genui-agent-service/api/v1/ws/tools/getDataCapabilitySchemas
```

输入：

```json
{
  "dataCapabilityIds": ["ViewWeather", "calendar.events.search"]
}
```

`dataCapabilityIds` 只能来自应用可选 `unavailableCapabilities` 过滤后仍可选的数据能力；字段缺失或为空数组时不额外排除。

输出：

```json
{
  "dataCapabilities": [
    {
      "id": "ViewWeather",
      "inputSchema": {},
      "outputSchema": {},
      "defaultWriteResultTo": "/data/weather"
    }
  ],
  "missingCapabilityIds": []
}
```

### generateWidgetCard

用途：首次生成和多轮编辑共用的主接口。

WebSocket path:

```text
/genui-agent-service/api/v1/ws/tools/generateWidgetCard
```

推荐入参：

```json
{
  "userQuery": "帮我做通勤卡片，包含天气和今日日程",
  "size": "2x4",
  "title": "通勤助手",
  "description": "天气日程速览",
  "candidateDataBindings": [
    {
      "capabilityId": "ViewWeather",
      "arguments": {
        "districtName": "青浦区",
        "forecastDays": 1
      },
      "writeResultTo": "/data/weather",
      "candidateOutputFields": [
        "/location/name",
        "/current/temperatureText",
        "/current/weatherText",
        "/daily/0/summary"
      ]
    },
    {
      "capabilityId": "calendar.events.search",
      "arguments": {
        "timeInterval": [1783238400000, 1783324799999]
      },
      "writeResultTo": "/data/calendar",
      "candidateOutputFields": [
        "/events/0/title",
        "/events/0/startTimeText",
        "/events/0/location"
      ]
    }
  ],
  "candidateEventCandidates": [
    {
      "capabilityId": "event.open.weather",
      "action": {
        "call": "clickToDeeplink",
        "args": {
          "intentName": "Weather_CityCode",
          "bundleName": "",
          "abilityName": "",
          "uri": "hww://www.huawei.com/totemweather?enterType=share&cityCode="
        }
      }
    }
  ],
  "candidateAssetIds": ["asset.weather.rain", "asset.calendar.schedule"]
}
```

说明：

- `sourceArtifactUrl` 为可选的上一版完整 artifact 永久 OBS URL。缺失表示首次生成；传入合法非空值表示基于该 artifact 编辑。首次生成不要传空字符串。
- edit 模式下，`userQuery` 只表达本轮修改需求；主 Agent 不下载来源 artifact，也不把历史 DSL 拼入请求。
- edit 模式下，省略 `size` / `title` / `description` 表示从来源 CardSpec 继承；显式传入时表示替换对应建议值并由微服务重新裁决或规范化。
- edit 模式下，省略某类候选数组表示从来源 artifact 继承并重新校验；显式传入数组表示编辑后该类别的完整候选集合，空数组表示清空该类别。
- 微服务只接受项目配置的 OBS host、bucket 和对象前缀，不得根据 `sourceArtifactUrl` 访问任意网络或本地地址。
- 来源 artifact 加载或校验失败时不得静默转为首次生成；编辑失败不得覆盖来源 artifact。
- 每次编辑必须生成新的完整 artifact 和新 URL，来源 artifact 保持不可变。
- `title` / `description` 是主 Agent 根据用户需求给出的静态展示文案建议，调用 `generateWidgetCard` 时必须传非空字符串，用于微服务生成最终可展示给用户的卡片标题和说明。
- 上述 `title` / `description` 必填规则适用于首次生成；edit 模式允许省略并继承来源 CardSpec。
- `title` 建议不超过 8 个字，`description` 建议不超过 12 个字；两者必须是稳定、静态、用户可理解的文案，不写动态值、隐私内容、未经确认的设备状态或能力可用性承诺。
- `title` / `description` 不替代 `userQuery`，不替代数据能力、事件能力或素材能力；微服务可以按最终能力裁决和降级结果规范化后写入最终 CardSpec 和 artifact 展示元信息。
- `title` / `description` 不进入 TaskSpec 顶层，避免扩展 A2UI 模型输入契约。
- 主 Agent 可见接口出参不暴露版本信息；版本仍必须记录在微服务日志和 artifact meta 中。
- `candidateDataBindings[].candidateOutputFields` 是主 Agent 根据能力 `outputSchema` 和用户 query 筛选出的候选展示字段路径，只表达展示倾向，不改变能力注册表中的原始 `outputSchema`。
- `candidateOutputFields` 是可选字符串数组；每一项都是 JSON Pointer，只允许引用对应能力 `outputSchema` 中存在的字段路径。字段类型、描述、示例值等由微服务根据能力注册表还原，不由主 Agent 传入。
- 不再传 `updateModel`；需要表达候选展示字段投影时只使用 `candidateOutputFields`。无法从 `outputSchema` 推导的路径不得传入。
- `candidateEventCandidates` 是事件候选单数组。
- 每个事件候选必须同时包含 `capabilityId` 和完整 `action.call/action.args`，参数可以是动态路径或静态值。
- 如果事件 action 无法从能力概述或用户明确输入中安全填齐，不传该事件候选。
- 微服务过滤某个事件能力时删除整个候选项，避免能力 ID 与事件参数错配。
- 本版主 Agent 不传 `slots`；生产默认不传 `options`。本地调试需要内联 artifact 时，才传 `options.returnArtifactInline`。
- 微服务默认 `allowDegradation: true`、`returnArtifactInline: false`。

输出：

```json
{
  "status": "success",
  "artifactUrl": "https://obs.xxx/widget/uuid.json",
  "artifactDigest": "sha256:xxx",
  "suggestSize": "2x4",
  "message": "已为你生成通勤卡片。",
  "removedCapabilities": [],
  "errorCode": ""
}
```

多轮编辑的完整开发指导见 `docs/多轮卡片编辑开发方案.md`。若专题文档与本方案冲突，以本方案为准。

## 生成接口状态码

`status` 只使用以下枚举：


## 能力过滤设计

微服务过滤顺序固定为：

```text
能力 ID 是否注册
 -> ROM/App 版本是否满足
 -> 依赖 App 是否安装且版本满足
 -> IDS 返回的意图/provider 是否存在
 -> arguments 是否符合 inputSchema
 -> writeResultTo 是否合法且无冲突
```

约束：

- App 安装只是必要条件，不是充分条件。
- 最终以 IDS 返回的意图列表、provider 列表和资源可用性为准。
- 不可用能力必须从最终 CardSpec 和 TaskSpec 候选中移除。
- 本期场景为一方 App，所以只有运动健康等少量一方应用可能会被卸载；其它能力也仍需以 IDS 和注册表校验为准。

## CardSpec 生成

主 Agent 传的是候选 `candidateDataBindings`。微服务过滤后得到 `effectiveDataBindings`，再生成最终 CardSpec。

动态 CardSpec：

```json
{
  "title": "通勤助手",
  "description": "天气日程速览",
  "suggestSize": "2x4",
  "dataBindings": [
    {
      "capabilityId": "ViewWeather",
      "arguments": {
        "districtName": "青浦区",
        "forecastDays": 1
      },
      "writeResultTo": "/data/weather"
    }
  ]
}
```

静态 CardSpec：

```json
{
  "title": "天气入口",
  "description": "快速打开天气",
  "suggestSize": "2x4"
}
```

规则：

- `title` / `description` 使用微服务最终规范化后的静态展示文案，用于端侧预览、添加确认和卡片管理等用户可见场景。
- `title` / `description` 不得包含动态数据值、真实隐私内容、未确认能力可用性承诺或内部字段名。
- `suggestSize` 使用微服务最终尺寸决策，不因静态卡片强制改为 `2x2`。
- 点击事件不进入 CardSpec，只进入 A2UI 模型的候选事件输入。
- 不可用能力不进入 CardSpec。
- 不编造 capabilityId。
- `writeResultTo` 必须位于 `/data/...`。
- 多个 `writeResultTo` 不得相同、互为父子或互相覆盖。
- `arguments` 必须符合能力 `inputSchema`。

## 端侧运行时数据刷新

本节明确 CardSpec、A2UI `DataModel` 和运行时 `updateDataModel` 的职责关系。CardSpec 不是 A2UI `DataModel` 的替代物，也不是另一份运行时状态；它是端侧可执行的数据获取与写入契约。A2UI `DataModel` 是 surface 的运行时状态容器，`updateDataModel` 是修改该状态的 A2UI 消息。

云侧生成阶段与端侧运行阶段的边界如下：

```text
云侧生成阶段
 -> 生成 genui DSL
    - createSurface
    - updateComponents
    - 初始 updateDataModel：只建立初始 DataModel 结构、空值和加载态
 -> 生成 CardSpec
    - dataBindings：声明 capabilityId、arguments、writeResultTo
    - 可选刷新计划：声明何时执行哪些 binding
 -> 将 genui + CardSpec 交付端侧

端侧运行阶段
 -> 读取 CardSpec.dataBindings
 -> 根据安装、可见、周期或其它已声明刷新条件触发 binding
 -> 调用 capabilityId 对应的端侧数据能力，并传入 arguments
 -> 按该能力的 outputSchema 保证、校验或归一化返回值结构
 -> 自动构造新的 A2UI updateDataModel 消息
    - surfaceId：目标卡片 surface
    - path：binding.writeResultTo
    - value：符合 outputSchema 的能力返回值
 -> 将消息注入 A2UI 运行时
 -> DataModel 变化触发绑定组件自动重渲染
```

以如下 binding 为例：

```json
{
  "capabilityId": "ViewWeather",
  "arguments": {
    "districtName": "青浦区",
    "forecastDays": 1
  },
  "writeResultTo": "/data/weather"
}
```

端侧调用 `ViewWeather` 后，若获得符合该能力 `outputSchema` 的返回值 `weatherResult`，则自动构造等价于以下形式的运行时消息：

```text
{
  "version": "v0.9",
  "updateDataModel": {
    "surfaceId": "实际卡片 surfaceId",
    "path": "/data/weather",
    "value": <weatherResult JSON 值>
  }
}
```

上例是消息构造示意，实际 `value` 是符合 `outputSchema` 的 JSON 值。`outputSchema` 由能力注册信息定义，CardSpec 无需重复复制完整 schema；云侧在生成和校验阶段使用该 schema 推导合法绑定路径，端侧数据能力框架在运行时据此保证或校验返回结构。

因此，云侧只需向端侧提供用于运行的 `genui + CardSpec`：云侧负责生成和静态一致性校验，不执行卡片安装后的数据能力调用，也不为每次刷新生成运行时数据。端侧负责刷新调度、能力调用、结果校验或归一化、`updateDataModel` 消息构造与注入。通过该边界，动态数据可以在端侧完成获取和消费，不需要经过卡片生成云服务。

## TaskSpec 构造

TaskSpec 是微服务传给 A2UI 模型的输入契约。`eventCandidates` 由微服务从过滤后的 `candidateEventCandidates` 派生。

顶层只允许：

```json
{
  "userQuery": "...",
  "size": "2x4",
  "eventCandidates": [],
  "dataModelSchema": {},
  "assetCandidates": []
}
```

不要在 TaskSpec 顶层加入：

```text
cardSpec
rules
capabilitySchemas
displayCandidates
role
component
layout
fontSize
dataModel
title
description
```

主 Agent 获取数据能力完整 `outputSchema` 后，可以根据用户 query 在 `candidateDataBindings[].candidateOutputFields` 中选择候选展示字段路径，减少无关字段进入后续生成链路。该字段筛选只作为候选投影，不改变能力注册表中的原始 `outputSchema`，最终仍由微服务校验、补齐和裁决。

`dataModelSchema` 由微服务根据最终 CardSpec、能力 `outputSchema`、主 Agent 候选字段投影和降级结果生成，是传给 A2UI 模型的参考数据结构。它应包含模型生成 DSL 所需的必要根结构、候选展示字段、字段类型、字段描述和实例值，不要求包含能力 `outputSchema` 的全部字段。

规则：

- `candidateOutputFields` 只由主 Agent 传字段路径，不传字段 `type`、`description` 或实例值。
- 微服务必须校验 `candidateOutputFields` 是否存在于对应能力 `outputSchema` 中，无法校验通过的字段不得进入 `dataModelSchema`。
- `type`、`description`、`sampleValue` 等字段说明由微服务根据能力注册表、受控样例和场景上下文还原或生成。
- `sampleValue` 表示给 A2UI 模型理解展示形态的实例值，优先使用接近 UI 展示的字符串，例如 `"26℃"`、`"多云"`、`"09:30 产品评审"`；它不代表用户真实运行时数据，也不改变 CardSpec 的动态数据契约。
- 对敏感或私密字段，`sampleValue` 必须使用脱敏样例或占位样例，不得使用真实用户数据。
- DSL 绑定路径仍必须绑定到动态数据路径，不能因为 `sampleValue` 存在而改写为静态文案。

```json
{
  "data": {
    "weather": {
      "location": {
        "name": {
          "type": "string",
          "description": "城市或区县名称",
          "sampleValue": "上海"
        }
      },
      "current": {
        "temperatureText": {
          "type": "string",
          "description": "当前温度展示文本",
          "sampleValue": "26℃"
        },
        "weatherText": {
          "type": "string",
          "description": "当前天气现象",
          "sampleValue": "多云"
        }
      }
    }
  }
}
```

A2UI 模型可以少用 `dataModelSchema` 中的候选字段，也可以新增受控的静态展示字段或文案辅助字段，但不得在动态数据能力输出路径下编造未授权字段，不得改变最终 CardSpec 所声明的数据能力契约。DSL 实际绑定到的动态路径必须能从最终 CardSpec、能力 `outputSchema` 或微服务允许的 `dataModelSchema` 中推导。

TaskSpec 示例：

```json
{
  "userQuery": "帮我做通勤卡片，包含天气和今日日程",
  "size": "2x4",
  "eventCandidates": [
    {
      "call": "clickToDeeplink",
      "args": {
        "intentName": "Weather_CityCode",
        "bundleName": "",
        "abilityName": "",
        "uri": "hww://www.huawei.com/totemweather?enterType=share&cityCode="
      }

  "dataModelSchema": {
    "data": {
      "weather": {
        "location": {
          "name": {
            "type": "string",
            "description": "城市或区县名称",
            "sampleValue": "上海"
          }
        },
        "current": {
          "temperatureText": {
            "type": "string",
            "description": "当前温度展示文本",
            "sampleValue": "26℃"
          },
          "weatherText": {
            "type": "string",
            "description": "当前天气现象",
            "sampleValue": "多云"
          }
        }
      },
      "calendar": {
        "events": [
          {
            "title": {
              "type": "string",
              "description": "日程标题",
              "sampleValue": "产品评审"
            },
            "startTimeText": {
              "type": "string",

            }

  "assetCandidates": [
    {
```

## A2UI 协议 profile

微服务根据 ROM/App/小艺版本选择协议 profile：

```text
romVersion + appVersion + xiaoyiVersion
 -> protocolProfileId
 -> 组件白名单、属性白名单、尺寸、绑定策略、事件策略、素材策略
```

第一期默认遵循：

- `2x2`: `160vp x 160vp`
- `2x4`: `320vp x 160vp`
- `createSurface.width/height` 和 root `styles.width/height` 必须写 `"matchParent"`；校验和内部布局预算按上述 profile 目标尺寸解析。
- 内部 Row/Column/Text/Image/Button/Progress 等继续使用可静态推导的数值宽高。
- `catalogId`: `ohos.a2ui.extended.catalog.form`
- `version`: `v0.9`
- 只允许组件：`Text`、`Image`、`Divider`、`Progress`、`Button`、`Checkbox`、`Row`、`Column`、`List`、`Stack`
- 禁止网络图、内联/base64 SVG data URI、未声明资源路径、未声明事件和禁用组件；允许来自有效素材候选或用户明确提供的本地/资源路径 SVG

绑定与表达式策略：

- 新生成 DSL 优先使用完整 `{{ ... }}` 表达式读取和拼接 DataModel；修复或评审已有 DSL 时，也接受协议声明的 `{"path":"/..."}` PathBinding 和宿主明确注册的 FunctionCall。
- 一个字符串只能包含一对完整表达式，不支持嵌套表达式；表达式内字符串使用单引号，内置函数只允许 `size()`。
- DataModel 推荐使用 JSON Pointer 片段，例如 `"{{ ${/data/weather/current/temperatureText} }}"`；同一卡片内优先保持同一种绑定风格。
- 表达式总长度不得超过 2048 字符，括号嵌套不得超过 20 层；求值失败返回空字符串，不得依赖失败态实现业务逻辑。
- `id`、`component`、对象 key、EventHandler `call`、`updateDataModel.path`、CardSpec `writeResultTo`、模板 `children.path` 和整个 `styles` 对象不允许使用表达式；单个允许动态值的组件属性、样式属性或事件参数可以使用表达式。
- `updateDataModel.path`、CardSpec `writeResultTo` 和模板 `children.path` 是结构 JSON Pointer，不得改写成表达式。

事件链策略：

- Form 只支持通用事件 `onClick`；事件值必须是非空 EventHandler 数组，并且每个事件只允许 1 个 handler。
- 每个 handler 必须包含 `call`；`call` 必须是有效事件候选中已经声明的函数标识符，不允许使用表达式。
- `args` 字段及嵌套结构必须符合对应事件能力的参数 schema 和合法目标；参数可使用安全静态值、完整表达式或协议允许的 PathBinding。
- 不支持 EventHandler `condition`、`as` 和 `$context` 事件上下文，不允许把多个动作串成未声明的行为链。
- 事件参数如果来自数据能力输出，必须能从最终 CardSpec 的 `writeResultTo + outputSchema` 或模板当前项中推导。

新增能力、组件或属性必须先进入 profile 和 validator，再进入 prompt。

## A2UI 模型输入与输出

微服务构造模型输入，不允许主 Agent 直接拼 prompt。

输入内容包括：

- 用户原始 query
- 目标尺寸
- TaskSpec
- 协议 profile 摘要
- 美观规则
- 不可用能力的降级上下文

A2UI 模型输出只接受一个 `genui` 代码块，且恰好三行 JSONL：

```genui
{"version":"v0.9","createSurface":{}}
{"version":"v0.9","updateComponents":{}}
{"version":"v0.9","updateDataModel":{}}
```

模型生成的组件动态值、素材和点击行为必须分别遵循当前 profile 的表达式策略、素材策略和事件链策略；不得根据通用 A2UI 知识自行扩展当前 Form 子集。

## Artifact

微服务上传完整 artifact，不只上传 DSL。

推荐结构：

```json
{
  "schemaVersion": "widget-artifact-v1",
  "genui": "三行 JSONL",
  "cardSpec": {},
  "taskSpec": {},
  "effectiveCapabilities": {
    "data": [],
    "event": [],
    "asset": []
  },
  "removedCapabilities": [],
  "meta": {
    "generationMode": "create",
    "protocolProfileId": "a2ui-form-rom7-v1",
    "capabilityRegistryVersion": "2026-07-03",
    "createdAt": 1780000000000
  }
}
```

端侧最少依赖：

- `genui`
- `cardSpec`

其中 `genui + cardSpec` 即为端侧渲染和运行时刷新的最小契约；端侧刷新链路不依赖 `taskSpec`、`effectiveCapabilities` 或其它云侧编排字段。其它字段主要用于排障、回放和灰度分析。

编辑生成的 artifact 建议在 `meta` 中记录 `generationMode: "edit"` 和 `sourceArtifactDigest`。不得记录完整永久 OBS URL。每次编辑上传新对象，不得覆盖来源 artifact。

## 校验与重试

校验对象是完整 artifact 文件，而不是只校验 DSL。

必查项：

```text
genui 恰好三行 JSONL
createSurface/updateComponents/updateDataModel 顺序正确
surfaceId 三行一致
catalogId 为 `ohos.a2ui.extended.catalog.form`，并与 profile 一致
createSurface 和 root 外围尺寸使用 `"matchParent"`，且按 size/profile 目标尺寸解析
CardSpec suggestSize 与 DSL 一致
组件、属性、事件、素材都在白名单内
children 引用可解析
完整表达式语法合法，长度和括号深度不超限，结构字段不使用表达式
PathBinding、FunctionCall 和表达式只出现在对应 schema 允许动态数据的位置
updateDataModel.value 不得在动态数据能力输出路径下新增未授权字段
DSL 绑定路径可从 TaskSpec.dataModelSchema、CardSpec outputSchema 或微服务允许的展示字段中推导
事件只使用 `onClick`，每个事件只有 1 个 handler，且不包含 `condition`、`as` 或 `$context`
事件 call/args 只能来自有效 eventCandidates，并符合对应参数 schema 和合法目标
图片只能来自有效 assetCandidates 或用户明确提供的本地资源；允许已声明的本地/资源路径 SVG，拒绝网络图、内联/base64 SVG data URI 和未声明路径
CardSpec 只包含有效 dataBindings
writeResultTo 不冲突
edit 模式的新 artifact URL 与来源 URL 不同
edit 模式已删除能力对应的绑定、事件和展示路径不再存在
```

失败后最多重新生成 1 次。重试 prompt 只追加错误摘要，不追加完整日志。

如果重试后仍失败：

- 可以降级为简单结构再生成一次，前提是仍在时延预算内。
- 否则返回 `failed`，并记录 `VALIDATION_FAILED` 或 `A2UI_GENERATION_FAILED`。

## 降级与拒答

主 Agent 看到的是候选能力，微服务才知道真实设备能力，所以主 Agent 不要提前说“可以生成天气日历卡”。推荐话术是：

```text
我先检查当前设备支持情况，然后为你生成可用的卡片。
```

处理规则：

```text
核心能力可用
 -> 正常生成

部分能力不可用
 -> 降级生成，说明少了什么

组件样式不支持
 -> 不拒答，改写为支持组件表达

数据能力不存在
 -> 不伪造动态 CardSpec，可做静态卡或入口卡

App 未安装
 -> 有入口或静态价值则降级，否则提示安装或启用

所有核心能力不可用且静态卡无价值
 -> 不调用 A2UI 模型，直接 unsupported
```

微服务内部状态需转为主 Agent 可感知的一系列描述：

```json
{
  "status": "degraded",
  "artifactUrl": "https://obs.xxx/widget/xxx.json",
  "message": "日历服务当前不可用，已先生成只包含天气的通勤卡片。",
  "removedCapabilities": [
    {
      "id": "calendar.events.search",
      "reason": "PROVIDER_NOT_FOUND",
      "userReadableReason": "日历服务当前不可用"
    }
  ]
}
```

示例 unsupported：

```text
当前设备上没有可用的外卖配送数据能力，也没有可打开的应用入口，所以暂时不能生成实时外卖卡片。你可以试试天气、日历或系统状态类卡片。
```

示例 degraded：

```text
日历服务当前不可用，我先为你生成只包含天气和通勤入口的卡片。日历服务恢复后可以再生成包含今日日程的版本。
```

## 美观质量

不要靠主 Agent 保证视觉质量。微服务通过 prompt、候选素材和校验器保证。

硬规则：

- 单一主焦点。
- `2x2` 最多 3 个主区域。
- `2x4` 最多 4 个主区域。
- 受保护文本不截断。
- 字号只使用批准阶梯：`10/12/14/16/18/20/32/40`。
- 间距只使用批准阶梯：`2/4/6/8/10/12/14/16`。
- root shell 必须完整：`width/height/padding/borderRadius/clip/background`；其中 root `width/height` 必须写 `"matchParent"`，内部布局仍按 profile 目标尺寸预算。
- Row/Column 宽高预算必须成立。
- 图片必须来自素材白名单，且设置明确 `width/height/objectFit`。
- 允许素材白名单中的本地/资源路径 SVG；禁止网络图、内联/base64 SVG data URI、未声明资源路径、emoji 和纯装饰堆叠。

生成失败或视觉风险高时，强制降级为：

```text
标题 + 主状态 + 1 条支撑信息 + 1 个入口
```

## OBS 与端侧标记

微服务上传完整 artifact，不只上传 DSL。

返回：

```json
{
  "status": "success",
  "artifactUrl": "https://obs.xxx/widget/uuid.json",
  "artifactDigest": "sha256:xxx",
  "suggestSize": "2x4",
  "message": "已为你生成通勤卡片。"
}
```

Skill 输出流式标记：

````text
```genWidgetResult:"https://obs.xxx/widget/uuid.json"```
````

端侧识别该标记后下载 artifact。不要在普通自然语言中伪造该标记。

端侧下载后拿到 `genui + cardSpec`，用于预览、确认添加和后续刷新。

当 `generateWidgetCard` 工具不可用、调用失败或返回产物不符合预期而进入主 Agent 兜底链路时，不输出 `genWidgetResult` 标记，除非主 Agent 真的完成 artifact 上传并拿到真实可下载 URL。

## 版本兼容

所有产物和关键配置必须带版本：

```text
apiVersion
taskSpecVersion
cardSpecVersion
dslProtocolVersion
skillVersion
protocolProfileId
capabilityRegistryVersion
artifactSchemaVersion
```

版本兼容要求：

- ROM < 7.0 不加载本 Skill。
- 微服务按 ROM/App/小艺版本选择 A2UI profile。
- 能力注册表支持最小 ROM、最小 App、最小包版本。
- artifact 记录生成时的 profile 和能力注册表版本。
- 新增能力、组件或属性必须先进入 profile 和 validator，再进入 prompt。

## 开发约束

开发代码时遵守：

- 正常链路不把最终 CardSpec 生成逻辑放到主 Agent。
- 例外：`generateWidgetCard` 工具不可用、调用失败或返回产物不符合预期时，主 Agent 可以进入兜底链路生成最终可交付结果；该路径必须显式标记为兜底，不得伪造动态能力或 artifact URL。
- 不把 A2UI 协议白名单散落在多个模块中；统一由 `A2UIProtocolRegistry` 提供。
- 不让 A2UI 模型自行选择未过滤能力。
- 不让 A2UI 模型生成或修改 CardSpec。
- 不把点击事件写入 CardSpec。
- 不让 A2UI 模型生成 `onClick` 以外的 Form 事件、多个 handler、`condition`、`as` 或 `$context` 事件上下文。
- 不把结构 JSON Pointer 改写为表达式，不在不支持动态数据的位置使用表达式、PathBinding 或 FunctionCall。
- 不使用网络图片、内联/base64 SVG data URI 或未声明资源路径；本地/资源路径 SVG 必须来自有效素材候选或用户明确输入。
- 不在 TaskSpec 顶层扩展非协议字段。
- 不在校验失败后无限重试。
- 不在能力不可用时伪造动态 dataBinding。
- 不上传半成品 artifact。
- 不只返回 OBS URL 而丢失结构化状态。
- 不让主 Agent 下载来源 artifact 或直接编辑 DSL。
- 不把 `sourceArtifactUrl` 直接放入 A2UI 模型 prompt。
- 不在 edit 模式来源加载失败时静默执行首次生成。
- 不覆盖任何来源 artifact；每轮编辑都上传新对象。

## 第一期开发任务

第一期最小闭环：

1. App 中控实现 ROM/App 版本分发，ROM < 7.0 不加载 Skill。
2. Skill 改造为工具编排型 SOP。
3. 微服务实现 3 个工具接口。
4. 能力注册表先用配置文件维护。
5. 接入 IDS 查询安装应用和意图列表。
6. 实现能力过滤、CardSpecBuilder、TaskSpecBuilder、PromptBuilder。
7. 接 A2UI 模型服务。
8. 实现规则校验器和 1 次重试。
9. 上传 OBS，返回 `genWidgetResult`。
10. 打通降级、拒答、失败状态和用户话术。
11. 在首次生成链路稳定后，按 `docs/多轮卡片编辑开发方案.md` 增加 `sourceArtifactUrl` 编辑链路。

这版不要一开始扩展过多数据能力或太细节的事项。先把一条动态卡片链路跑稳，再逐步补能力。先保证链路稳定、能力不伪造、卡片能渲染、失败能给用户合理回复。

## 最小测试集

至少覆盖：

```text
帮我做一个通勤日常卡片，包含今天天气和今日会议
帮我做一个只显示今天上海天气的桌面卡片
帮我做一个今日会议提醒卡片，可以点击查看日程
帮我做一个低电量省电卡片
帮我做一个雨天打车提醒卡片
帮我做一个睡眠监督卡片
帮我做一个美团外卖配送状态卡片
帮我做一个可以输入文字的备忘录卡片
帮我做一个包含天气、日历、新闻、股票的大卡片
帮我做一个打开天气应用的入口卡片
把刚才的卡片背景改成蓝色
把刚才卡片里的日历去掉，只保留天气
把上海天气改成北京天气
把刚才的卡片从 2x2 改成 2x4
```

测试维度：

- 正常动态卡片。
- 部分能力不可用后的 degraded。
- 全部核心能力不可用后的 unsupported。
- App 未安装。
- 素材不可用。
- A2UI 模型输出非法 DSL。
- `catalogId` 不是 `ohos.a2ui.extended.catalog.form`。
- 已声明本地 SVG 正常通过；网络图、内联/base64 SVG data URI 和未声明 SVG 被拒绝。
- 完整表达式、PathBinding 和已注册 FunctionCall 正常通过；嵌套表达式、超限表达式和结构字段表达式被拒绝。
- `onClick` 单 handler 正常通过；其它事件、多 handler、`condition`、`as`、`$context` 和未声明 call/args 被拒绝。
- OBS 上传失败。
- `writeResultTo` 冲突。
- 端侧能够按 CardSpec `dataBindings` 调用正确的数据能力，并将 `arguments` 原样映射到能力入参。
- 端侧能够拒绝或归一化不符合能力 `outputSchema` 的返回值，不把非法结构写入 DataModel。
- 端侧能够以 `writeResultTo` 为 `path`、以能力返回值为 `value` 自动构造运行时 `updateDataModel` 消息，并注入正确的 surface。
- 运行时 `updateDataModel` 更新后，绑定对应 DataModel 路径的组件自动重渲染，且无需重新请求云侧生成 DSL 或 CardSpec。
- 端侧运行时数据刷新不依赖将能力返回值上传至卡片生成云服务。
- 老版本 ROM 不加载 Skill。
- 不传 `sourceArtifactUrl` 时首次生成保持兼容。
- 纯视觉编辑继承 CardSpec 和全部有效能力。
- 编辑模式省略候选数组、传完整数组和传空数组的语义正确。
- 来源 URL 非法、越权、下载失败、schema 不支持或 artifact 损坏。
- 编辑时重新进行设备能力裁决。
- 编辑成功生成新 URL且不覆盖来源 artifact。
- 连续多轮编辑始终使用上一轮返回的新 URL。
