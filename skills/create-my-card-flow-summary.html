Create My Card 链路改造方案

面向小艺 App 中“自然语言生成鸿蒙桌面卡片”的工程化升级：把主对话 Agent 的通用生成，改造为 Skill 编排 + 专用小模型生成 + 文件产物校验 + 端侧渲染的闭环。

优先目标：效果稳定 兼顾目标：端到端时延 产物：GenUI DSL + CardSpec 端侧：A2UI Form 渲染

一、为什么要改造

效果

让模型专门学会生成卡片协议

DSL/CardSpec 是强格式、强约束任务。专用小模型可以围绕 Form 组件、数据绑定、CardSpec 对齐和常见卡片布局做增训，减少通用模型偶发的协议漂移。

时延

避免大模型复述小模型结果

小模型已以 token 形式生成完整产物后，如果再让主 Agent 复述给端侧，会产生二次 token 输出成本。文件产物可以被校验、发布和端侧直接拉取。

工程化

把正确性交给校验器

语法、组件白名单、数据路径、能力 manifest、CardSpec 对齐等不应只靠 prompt 约束。校验程序能在渲染前拦住失败产物。

二、新旧链路对比

当前链路更轻，但生成、校验、端侧传输都耦合在主 Agent 输出中；目标链路把生成产物文件化，让校验、修复、缓存和端侧消费都有稳定边界。

当前链路

Query 用户输入自然语言需求
Skill 读取规则并直接生成 DSL/CardSpec
Token 输出 以文本流形式输出 GenUI DSL
端侧解析 从 token 流恢复消息
渲染 A2UI Form 展示卡片

目标链路

Query 用户原始需求
Skill 精炼 生成 CardTaskSpec
能力路由 按需选取 manifest
小模型服务 生成 DSL + CardSpec
产物文件 保存 artifact
校验器 语法与协议检查
发布链接 文件 ID / URL / hash
端侧渲染 拉取并解析 A2UI

三、核心变化点

变化项 当前方案 目标方案 意义与价值
生成主体 Skill 通过大模型直接生成 DSL/CardSpec。 Skill 负责编排，小模型服务专门生成卡片产物。 生成能力可以单独增训、评测、灰度和版本化。
输入形态 原始 query + 当前 Skill 规则。 原始 query 被精炼为 CardTaskSpec，再传给小模型。 减少无关上下文，降低 prompt 长度和推理时延。
产物形态 DSL/CardSpec 以 token 形式输出。 保存为统一 artifact 文件，再给端侧文件链接。 便于校验、修复、缓存、复用和端侧稳定消费。
正确性保障 依赖模型自检和 prompt 约束。 引入校验程序，失败后结构化修复。 把可程序化判断的问题从模型概率输出中剥离出来。
端侧消费 端侧从 token 流中解析 DSL。 端侧按 artifactId/URL/hash 拉取产物文件。 减少二次转述，支持缓存和失败重试。

四、传给小模型的信息边界

原则：小模型只拿生成本张卡片必须知道的信息，不接收小艺主 Agent 的完整 system prompt，也不接收完整 Skill 文档。

建议传入
  • 用户原始 query。
  • Skill 精炼出的 CardTaskSpec。
  • 协议版本、catalogId、目标尺寸。
  • 本次相关 data capability manifest。
  • 本次相关 event capability manifest。
  • 少量语义匹配素材候选。
  • 当前日期、时区、语言区域。
不建议传入
  • 小艺主对话 Agent 的完整 system prompt。
  • 完整 Skill 文档和全部 reference。
  • 与本次卡片无关的能力 manifest。
  • 长历史对话，除非用户明确引用。
  • 端侧不支持或未声明的自定义函数。

CardTaskSpec 示例

{
  "query": "帮我做一个通勤日常桌面组件，包含今天天气、今日待办清单",
  "task": {
    "scene": "commute_daily",
    "sizePreference": "2x4",
    "mustShow": ["today_weather", "today_schedule"],
    "dataCapabilities": ["weather.overview.get", "calendar.events.search"],
    "eventCapabilities": ["clickToDeeplink", "clickToIntent"],
    "locale": "zh-CN",
    "timezone": "Asia/Shanghai",
    "currentDate": "2026-06-27"
  },
  "runtime": {
    "protocolVersion": "v0.9",
    "catalogId": "ohos.a2ui.extended.catalog",
    "targetFormFactors": ["2x2", "2x4"]
  }
}

五、文件产物格式建议

建议不要把 GenUI DSL 和 CardSpec 分成两个互不关联的文件，而是保存为一个卡片 artifact。这样校验器可以一次性检查 DSL、CardSpec、DataModel 路径和能力声明是否对齐。

{
  "artifactVersion": "1.0",
  "protocolVersion": "v0.9",
  "surfaceId": "commute-daily-card",
  "genui": [
    {"version": "v0.9", "createSurface": {}},
    {"version": "v0.9", "updateComponents": {}},
    {"version": "v0.9", "updateDataModel": {}}
  ],
  "cardspec": {
    "suggestSize": "2x4",
    "dataBindings": []
  },
  "meta": {
    "sourceQuery": "...",
    "skillVersion": "...",
    "modelVersion": "...",
    "createdAt": 1782576000000
  }
}

六、当前 Skill 需要做的适配

编排职责
  • 新增 CardTaskSpec 生成步骤。
  • 按 query 选择相关 data/event capability。
  • 控制在线小模型服务调用。
  • 管理校验、修复、发布流程。
规则调整
  • 修改“不得创建产物文件”的旧约束。
  • 增加 artifact 文件格式约束。
  • 保留现有 Form 协议与 CardSpec 规则。
  • 把最终交付从代码块改为文件链接或文件 ID。
修复闭环
  • 定义校验脚本入参：artifact 文件路径。
  • 定义校验脚本出参：结构化错误列表。
  • 校验失败后最多重试 1 到 2 次。
  • 连续失败时降级为能力边界说明。

七、小模型服务侧需要做的适配

System Prompt

预置稳定硬规则：组件白名单、协议版本、JSON 输出、表达式禁用、CardSpec 对齐、禁止伪造能力。

增训数据

覆盖常见卡片类型、不同尺寸布局、query 到 CardTaskSpec 到 artifact 的样例，以及校验失败修复样例。

服务接口

输入 CardTaskSpec 和精简 manifest，输出 artifact 文件路径、artifactId、模型版本和生成状态。

修复接口

输入原 artifact 和结构化校验错误，返回修复后的 artifact，避免从零重新生成。

八、校验程序建议

校验层级 检查内容 失败示例
语法校验 artifact 是否为合法 JSON，genui 每条消息是否可解析。 JSON 缺括号、数组尾逗号、字符串未闭合。
协议校验 version、catalogId、surfaceId、组件白名单、属性名、事件类型。 使用 Text.text、Image.url、Grid、onChange。
CardSpec 校验 suggestSize、capabilityId、arguments、writeResultTo、能力 manifest。 使用未声明的 todo capability，writeResultTo 不在 /data 下。
路径一致性校验 DSL 中 path 能否从 updateDataModel 或 writeResultTo + outputSchema 推导。 UI 绑定 /data/todo/items，但 CardSpec 没有 todo binding。
布局静态校验 尺寸、主区域数量、Row 子节点数、关键文本 ellipsis、网络图路径。 2x2 中出现长列表，CTA 使用 ellipsis。

ValidationResult 示例

{
  "valid": false,
  "errors": [
    {
      "code": "UNDECLARED_CAPABILITY",
      "severity": "blocking",
      "path": "/cardspec/dataBindings/1/capabilityId",
      "message": "capabilityId todo.items.search is not declared in manifest"
    }
  ],
  "warnings": [
    {
      "code": "DENSE_ROW",
      "severity": "warning",
      "path": "/genui/1/updateComponents/components/eventRow",
      "message": "Row contains more than 3 direct children"
    }
  ]
}

九、分层放置规则

内容类型 建议放置位置 原因
稳定协议硬规则 小模型 system prompt + 校验器 减少每次请求传输成本，同时由程序兜底。
常见卡片布局经验 小模型增训数据 让模型内化效果，而不是靠超长 prompt 临时约束。
变化快的能力 manifest Skill 按需注入 能力会扩展，不能固化在模型权重或长 prompt 中。
用户 query 理解与业务路由 当前 Skill Skill 更适合读取上下文、做任务精炼和选择工具链。
语法、路径、能力一致性 校验程序 这些问题可确定性判断，不应主要依赖模型自觉。

十、推荐推进节奏

阶段 1

定义 artifact 与校验器 MVP

先支持 JSON 语法、Form 组件白名单、CardSpec suggestSize、dataBindings 和 path 一致性校验。此阶段可以继续由当前 Skill 生成产物。

阶段 2

引入在线小模型服务

Skill 生成 CardTaskSpec，调用小模型生成 artifact 文件。先覆盖天气、日历、会议提醒等高频场景。

阶段 3

加入修复循环和发布链路

校验失败后返回结构化错误，由小模型修复文件。通过后生成文件链接、artifactId 和 hash，供端侧拉取。

阶段 4

扩展能力与评测体系

补充待办、通勤、位置、路线、刷新计划等 manifest，并建立效果评分、渲染成功率、修复成功率和端到端时延指标。

十一、结论

目标链路的本质是职责拆分：Skill 负责理解、路由和编排；小模型负责高质量生成；校验器负责确定性正确；端侧负责拉取、执行数据能力并渲染。这样既能提升卡片效果，也能减少重复 token 输出带来的时延，并为后续扩展更多端侧能力留下清晰接口。

建议后续补齐：todo data capability、commute/location data capability、CardSpec refreshPlan、权限与失败态契约、artifact 发布与缓存策略。
