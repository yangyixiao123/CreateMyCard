# 设计评审

形成 `genui` 与 `cardspec` 草稿后使用本文档。所有修复都必须使用 HarmonyOS A2UI Form 样式和组件，并直接更新当前上下文中的草稿。

## 紧凑卡片检查

- 卡片回答一个一眼可读的问题。
- 选择的尺寸明确：`2x2`（`160 x 160vp`）或横版 `2x4`（`320 x 160vp`）。
- `2x2` 主视觉区域不超过 3 个。
- `2x4` 主视觉区域不超过 4 个。
- 主指标/标题明显占主导。
- 次要上下文短于主内容。
- 动作明显，但不大于主信息。
- 没有长正文、文章章节、表格和重型列表。
- 卡片从可泛化规则构造，而不是从模板/示例产物复制。

## 布局检查

- Root 有卡片式 shell：圆角、裁剪，以及清晰背景或图片。
- 只有一个外层卡片 shell；内部分组不要变成多个互相竞争的小卡片。
- 外层圆角应不小于内层圆角。
- 阴影主要属于 root 或一个聚焦浮动动作，不属于每个小块。
- Root 尺寸适配所选目标。如果 width 是 `"100%"`，内部布局仍然遵守 `2x2` 的 `160vp` 或横版 `2x4` 的 `320vp`。
- 横向行有 2 或 3 个直接子节点，除非使用有明确理由的重复项模式。
- 时间、CTA、状态标签、价格或百分比等受保护文本没有使用狭窄固定宽度。
- 弹性文本列使用 `layoutWeight`；小图标和固定圆形使用明确宽高。
- `maxLines` 和 `minFontSize` 用于保护受约束文本，同时不隐藏关键信息。
- 关键信息不能靠截断“保护”。日期、星期、时间、状态、CTA、主指标、主标题和用户要求字段必须完整显示，而不是用 ellipsis、clip 或 marquee。
- 拥挤 Row 有明确宽度预算；分隔线、间距、padding 和固定列没有占用受保护文本所需空间。

## 视觉检查

- 色板贴合当前场景，并且是为当前场景选择的。
- 渐变和光晕层支持内容，而不是纯装饰。
- 渐变或图片上的文字对比足够强。
- 半透明块有边框或背景对比。
- 如果使用图片，它是真实本地/资源路径，且重要主体没有被文字遮住。
- 如果素材库中存在与当前场景语义匹配且能提升表达的素材，应优先使用素材库素材；如果使用素材库素材，它的 `description` 必须匹配当前语义角色。
- 如果图片不能增加场景特定信息，应改用非媒体视觉技法。
- `2x2` 只有一个主导视觉锚点。
- `2x4` 有一个主导视觉锚点，最多一个次级视觉装置。

## 交互检查

- 可点击视觉区域有 `onClick` EventHandler。
- `onClick.call` 优先来自 event capability 的 `functionCall`；宿主假设函数名必须有意义且稳定。
- 必要动作参数从 DataModel、事件上下文或 event capability 允许的静态目标取得；业务 ID 不硬编码。
- 跳转目标在 event capability 的 `supportedTargets` 中，事件参数符合对应 `parameters`。
- 主指标、动作和有意义的纯图标区域有可见文本或相邻语义文本说明，不依赖未声明的 accessibility 属性。

## 数据检查

- DataModel 键是语义化的，并按领域分组：`weather`、`meeting`、`action`、`asset` 等。
- 不要留下示例或历史输出中的样例值。
- 当本地化、格式化或多值运算复杂时，预先计算展示字符串，不使用表达式。
- 没有网络/占位媒体 URL、未声明资源路径，也没有从示例遗留的业务样例事实。

## 最终检查

任何编辑后都重新检查受影响的视觉、布局、交互和数据评审项，然后回到 [`final-review.md`](final-review.md) 完成最终评审。
