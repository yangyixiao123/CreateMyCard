# Harmony 卡片模板生成方案设计

## 1. 背景、目标与非目标

本方案为主 Agent 提供一条模板驱动的 HarmonyOS A2UI Form 卡片生成路径。目标是把卡片布局的自由度收敛到经过预算的 region、内容槽位和组件变体，使同类需求得到稳定、美观且可校验的 `genui` 与 `cardspec`。

本方案不负责云侧 artifact 生成，不调用微服务，也不复用其它卡片生成 Skill 的运行时上下文。它不支持完整 App 页面、复杂表单、长报告、任意尺寸或跨模板自由拼装。

## 2. 三个 Skill 的关系

三个 Skill 是独立方案：

| Skill | 方案 | 输出 |
| --- | --- | --- |
| `harmony-card-generation-online` | 云侧工具编排 | artifact 链接和状态说明 |
| `harmony-card-generation-offline` | 通用规则自由生成 | DSL 与 CardSpec |
| `harmony-card-template-generation` | 固定模板与受控变体生成 | DSL 与 CardSpec |

它们互不调用、互不兜底。新 Skill 通过显式 `$harmony-card-template-generation` 使用，并关闭隐式召回。

## 3. 模板来源

同级 `template/` 中的十组 `.dat` 和图片只提供历史视觉观察。现有 offline 的九个规范化 `2x2` 模板提供首版骨架来源。两个历史 `2x4` 布局用于提取宽卡分区关系，另外两个宽卡从主状态、日程和多指标场景推导。

来源文件不成为运行时依赖。历史文件中的旧 catalog、固定画布写法、`theme`、emoji、内联/base64 SVG、网络图、未声明素材、旧事件和不合规属性均不复制；素材库声明的本地 SVG/PNG 可用。

## 4. 模板资源职责

- `assets/templates/index.json`：维护模板顺序、尺寸、路由特征和文件路径。
- `assets/templates/manifest.schema.json`：约束所有 manifest 的公共形态。
- `manifest.json`：锁定布局、region、槽位、组件变体、风格和删减规则。
- `template.genui.jsonl`：提供默认合法组件树和稳定几何起点。
- `cardspec.json`：提供静态 CardSpec 骨架；动态绑定由当前需求补充。

主 Agent 每次只读取一个模板目录，不同时比较多个完整骨架。

## 5. 路由与组件选择

先把需求拆成一个服务对象、一个主问题以及主显示、支撑、并列事实、动作和素材角色。模板选择先按尺寸、必选槽位、动作数、数据能力数、文本预算和素材可用性做硬过滤，再按语义、风格、删减量和索引顺序确定唯一模板。

region 的宽高和层级固定。组件只能在 manifest 声明的同尺寸变体之间切换，例如动作区在文字按钮、图标文字行或静态支撑之间切换。主视觉类型不允许跨语义替换：进度模板保留 Progress，产品模板保留 Image，倒计时模板保留 Text 主值。

详细字段见 [模板契约](references/template-contract.md)，路由和风格见 [路由与风格](references/routing-and-style.md)。

## 6. 首版模板

### 2x2

九个模板使用 `160x160vp` 预算、12vp padding、136x136vp 内容区和 18vp root 圆角：

| 模板 | 主要结构 | 典型场景 |
| --- | --- | --- |
| `meter-side-action` | 标题、仪表、短事实、动作 | 电池、睡眠、进度 |
| `multi-meter-action` | 多个小状态、主指标、动作 | 内存、设备健康 |
| `bar-dual-metric-action` | 进度条、双指标、动作 | 应用时长、阈值 |
| `countdown-panel` | 标题、主数字、支撑面板 | 倒计时、目标 |
| `meeting-action-badge` | 日程、元数据、徽标、动作 | 会议、预约 |
| `product-stat-tiles` | 产品视觉、状态、双 tile | 耳机、穿戴设备 |
| `quadrant-ambient-action` | 四象限角色布局 | 天气关怀、快捷动作 |
| `spotlight-context-action` | 主值、上下文面板、动作 | 天气、通勤 |
| `stacked-schedule-action` | 纵向日程与动作 | 下一日程、专注 |

### 2x4

四个模板使用 `320x160vp` 预算、12vp padding、296x136vp 内容区和 18vp root 圆角：

| 模板 | 固定区域 |
| --- | --- |
| `content-action-sidebar` | 24vp 标题；下方 `200 + 12 + 84vp` 内容与侧栏 |
| `dual-action-panel` | `180 + 12 + 104vp` 内容与操作区；操作区为两个 62vp 动作 |
| `hero-context-action` | `104 + 12 + 180vp` 主视觉与上下文；右栏为 96vp 信息、8vp 间距、32vp 动作 |
| `metric-dashboard-action` | `92 + 8 + 124 + 8 + 64vp` 主指标、多指标与短动作/状态 |

## 7. 数据、事件、素材与 CardSpec

`2x2` 最多一个数据能力，`2x4` 最多两个。能力必须来自 [数据能力索引](references/capability/data-capability/index.md)，UI 路径必须由 `writeResultTo + outputSchema` 推导。未声明能力只能做静态降级。

点击事件只写 DSL `onClick`，不进入 CardSpec；事件目标必须来自 [点击事件能力](references/capability/event-capability/click-event.md)。素材只使用 [素材库](references/design/asset-library.md) 声明的本地路径。

CardSpec 只维护静态标题、静态描述、推荐尺寸和数据绑定，详细约束见 [CardSpec 契约](references/capability/cardspec.md)。

## 8. 校验、修复与无匹配处理

生成结果依次经过协议、组件、CardSpec、表达式、素材、绑定、跨文件、颜色、质量和模板一致性检查。首次失败后只允许按诊断修复一次；第二次仍失败时不输出产物。

无模板匹配时依次删除可选内容、尝试 `2x4`、使用同尺寸安全通用模板。仍不能满足受保护文本、动作或能力约束时，要求用户简化需求，不调用其它 Skill，也不自由生成。

## 9. 测试与扩展

首版测试覆盖十三个骨架、四种风格、六类数据能力、点击与素材，以及文本超限、能力缺失、素材缺失、事件缺失和信息过载。所有模板必须通过 `validate_templates.py --all --strict`，生成草稿必须通过 `validate_card.py --template-id ... --strict`。

新增模板时必须先说明独立布局价值，避免与已有模板重复；随后补充 manifest、双骨架、索引项、路由样例和测试。禁止只复制历史 `.dat` 后加入索引。

## 10. 版本与同步

本文件是模板生成方案的维护入口。模板契约、路由算法、风格档案、输出约束或校验策略变更时，先修改本文件，再同步 `SKILL.md`、references、模板、脚本和测试。

共享 A2UI 协议和端侧能力边界仍服从项目权威设计；若本文件与项目方案冲突，先确认再修改实现。
