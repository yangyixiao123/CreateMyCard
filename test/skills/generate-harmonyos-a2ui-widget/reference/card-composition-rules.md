# 可泛化卡片构造规则

本文档替代基于模板的生成。它定义 `2x2` 与 `2x4` HarmonyOS GenUI 桌面卡片的可复用构造规则。

## 构造心法

从用户含义出发，而不是从示例产物出发。

1. 提取语义角色。
2. 选择 `2x2` 或 `2x4`。
3. 分配尺寸预算。
4. 用简单语法组合区域。
5. 按语义绑定数据。
6. 先校验，再评审视觉质量。

## 语义角色

每张生成卡片在写 DSL 前，都应把用户内容分配到这些角色：

| 角色 | 含义 | 常用组件 |
| --- | --- | --- |
| `identity` | 这张卡片关于什么：标题、设备名、人物、地点、事件 | `Text`、小 `Row` |
| `primaryAnswer` | 最重要的一眼可见结果 | 大 `Text`、`Progress`、图片焦点 |
| `metric` | 数字、时间、温度、电量、价格、时长 | 受保护 `Text`、`Progress` |
| `context` | 次要说明或临近事件/详情 | 紧凑 `Column`、`Row`、`Divider` |
| `status` | 良好/警告/错误/模式标签 | 带背景、字形或强调色的小 `Text` |
| `media` | 用户提供的真实本地/URL 素材 | `Stack` 内的 `Image` |
| `action` | 一个下一步操作 | 可点击 `Stack`/`Row`/`Column` 或 `Button` |

规则：

- `primaryAnswer` 必须在视觉上占主导。
- `context` 必须短于 `primaryAnswer`。
- `action` 支持卡片，不应成为主视觉，除非用户要求纯动作卡。
- 如果没有可靠 `media`，使用渐变、半透明块、字形文本、`Divider` 或 `Progress`。

## 尺寸决策规则

使用 `2x2` 当：

- 卡片有一个主答案，最多一个上下文组。
- 用户要求紧凑 widget 或单一状态/提醒。
- `160 x 160vp` 布局可以无滚动容纳。

使用 `2x4` 当：

- 卡片需要 2 到 3 个紧凑辅助事实。
- 场景天然存在左右关系：媒体加状态、主指标加详情、日程加动作、关怀状态加联系。
- 受保护关键信息需要比 `2x2` 更安全的横向空间。
- 主视觉需要呼吸空间，并且动作/上下文在 `160vp` 高度内仍然重要。

以下情况不要用 `2x4`，而应升级说明：

- 需要超过 4 个主要区域。
- 内容需要段落、表格、表单或长动态列表。
- 用户真正想要详情页。

## `2x2` 规则：`160 x 160vp`

### 硬预算

- Root 目标：`width: 160`，`height: 160`。
- Root padding：`10` 到 `12`。
- 可用内部区域：约 `136 x 136vp`。
- 主区域：<= 3。
- 直接 `Row` 子节点：<= 3。
- 不滚动。
- 默认不使用动态列表模式。

### 区域预算

选择一种分配节奏：

| 节奏 | 使用场景 | 预算 |
| --- | --- | --- |
| `header-primary-action` | 提醒、事件、CTA 卡 | header `18-24`，primary `58-72`，action `30-36` |
| `primary-context-action` | 带详情的状态/指标 | primary `58-76`，context `34-44`，action `28-34` |
| `visual-primary-context` | 进度/设备/健康速览 | visual `56-76`，primary/context 分享剩余高度 |
| `single-focal` | 一个强结果，可选极小 meta | focal `88-110`，meta/action `22-34` |

这些预算是指导，不是精确模板。用它们避免文本在 `160vp` 内互相碰撞。

### 组合语法

使用一个 root shell：

```text
RootShell(Column|Stack)
  -> Identity?
  -> PrimaryAnchor
  -> ContextStrip?
  -> ActionPill?
```

允许的变化：

- 使用图片背景、光晕或进度叠层时，root 可用 `Stack`。
- 以文本/指标/动作为主时，root 可用 `Column`。
- 只有当区域内部表达真实关系时才用 `Row`：图标 + 标签、指标 + 状态、进度 + 数值。

### 文本预算

- 标题/identity：1 行。
- 主值：1 行；如果值可能变化，使用 `minFontSize`。
- 上下文：1 行；如果没有动作，可用 2 行非常短的文本。
- CTA：1 行；按内容宽度或全宽 pill。
- 避免任何像段落的正文。

### 视觉规则

- 只保留一个视觉锚点：大指标、进度环、图片主体、图标/字形或状态徽章。
- 最多 1 个主要半透明块。
- 只有当图片真实且有用时才使用图片。否则使用非媒体视觉技法。
- 保持强对比；图片上的文字需要遮罩或安全位置。

## `2x4` 规则：`320 x 160vp`

### 硬预算

- Root 目标：`width: 320`，`height: 160`。
- Root padding：`10` 到 `12`。
- 可用内部区域：约 `296 x 136vp`。
- 主区域：<= 4。
- 直接 `Row` 子节点：<= 3。
- 默认不滚动。如果静态内容放不下，收敛内容。

### 区域预算

选择一种横向节奏：

| 节奏 | 使用场景 | 预算 |
| --- | --- | --- |
| `focal-detail-action` | 通用丰富紧凑卡 | focal pane `108-132w`，detail pane `120-150w`，action/status `40-56w` 或底部条 |
| `media-state-action` | 带真实图片的产品/设备/地点 | media pane `112-140w`，state/details `120-150w`，action `40-56w` |
| `metric-context-action` | 天气、进度、健康、财务 | metric pane `96-124w`，context pane `132-164w`，action/status `44-60w` |
| `sequence-detail-action` | 日程/计划/关怀卡 | sequence pane `132-164w`，detail pane `92-124w`，action/status `44-60w` |

2x4 卡片可以有两个并排 2x2 区域的感觉，但必须仍是一个卡片 shell 和一个层级。

### 组合语法

```text
RootShell(Row|Stack)
  -> FocalPane
  -> DetailPane
  -> ActionOrStatus?
```

允许的变化：

- 每个 pane 内使用 `Column` 形成竖向节奏。
- 当真实图片/背景或光晕层跨越整张卡时，root 使用 `Stack`。
- 只有在受保护文本仍可读时，才把 `ActionOrStatus` 放为右侧栏、底部条或最后一个详情行。
- 使用 2 到 3 行静态详情簇；除非用户明确需要可变重复数据，否则避免动态列表。

### 文本预算

- Focal pane：1 行主值/标题，加 1 个可选 caption。
- Detail pane：2 到 3 个紧凑行，每行 1 行。
- 建议/正文：总计 <= 2 行短文本。
- CTA/status：1 行，受保护；除非已知完整标签能放下，否则避免窄固定 CTA。

### 视觉规则

- 一个主导视觉锚点，最多一个次级视觉装置。
- 最多 2 个主要半透明块。
- 进度环、媒体主体或大数字可以占据左 pane。
- 右 pane 应承载详情和动作，不要再放一个竞争性的 hero。

## 场景到结构映射

这些是规则，不是模板。先按语义角色选择结构，再为场景创建新的 ID、数据键、颜色和间距。

| 场景 | 优先尺寸 | 结构逻辑 |
| --- | --- | --- |
| 天气 + 下一件事 | `2x2` | 主天气指标 + 下个事件上下文 + 动作/状态 |
| 会议/提醒 | `2x2` | identity 标签 + 受保护时间/标题 + 地点/动作 |
| 倒计时 | 单个事件用 `2x2`，事件 + 小计划用 `2x4` | 大倒计时 + 单位 + 辅助计划/动作 |
| 健康/进度 | 单指标用 `2x2`，指标 + 趋势用 `2x4` | 进度视觉 + 时长/数值 + 提醒/建议 |
| 家庭关怀告警 | 如果建议/联系/详情重要，用 `2x4` | 状态/天气 + 风险详情 + 联系动作 |
| 设备/产品状态 | 简单电量/模式用 `2x2`，媒体 + 状态用 `2x4` | 真实素材或字形 + 电量/模式行 + 动作 |
| 信息摘要 | 极小摘要用 `2x2`，属性 + 短上下文用 `2x4` | 标题 + 属性 + <=2 行摘要 + 动作 |

## 宽度与换行规则

宽度预算严格：`2x2` 宽 `160vp`，横版 `2x4` 宽 `320vp`。两者高度都只有 `160vp`。

- 只有在值长度已知且完整字符串能放下时，才给受保护值设置显式宽度。
- 文本列紧邻固定图标或仪表时，应使用 `layoutWeight: 1`。
- 如果一行同时包含指标和句子，句子是可压缩内容。
- 对受约束的关键文本使用 `maxLines: 1` 和 `minFontSize`。`textOverflow: "ellipsis"` 只用于可选次要文本。
- 不要把短中文标签、单位、时间、价格、温度或 CTA 标签按字符拆开。

## 关键信息完整显示规则

除非用户明确接受截断，否则关键信息必须完整可见。

默认把这些视为关键信息：

- 日期、星期、时间、倒计时、时长、温度、价格、评分、电量、百分比和主指标值。
- CTA 标签、状态徽章、告警标签、短中文短语和主事件/物品标题。
- 用户明确要求卡片显示的任何字段。

规则：

- 不要在关键 `Text` 组件上设置 `textOverflow: "ellipsis"`。
- 不要把 ellipsis 改成 `clip` 来修关键文本；裁剪仍然会隐藏信息。
- 如果关键文本横向放不下，写 JSON 前先重新平衡布局：
  1. 减少装饰性 padding 和 `itemMargin`。
  2. 减少固定列和分隔线厚度/宽度。
  3. 在保持层级的前提下降低字号。
  4. 把可压缩次要文本移动到另一行，或从演示数据中移除。
  5. 把该行拆成 `Column`、选择横版 `2x4`，或在请求无法容纳时升级说明。

## Row 宽度预算

生成任何拥挤 Row 前，在布局理由中写出计算：

```text
available = parentWidth - horizontalPadding - totalItemMargins - fixedIconOrDividerWidths - fixedTextWidths
```

然后检查每个受保护字符串能否放入分配宽度。对 `2x2` 卡片，假设 root padding 后父级内容宽度通常为 `136vp`。在嵌套半透明块内部，padding 和行间距很容易让可用宽度低于 `120vp`；当多个受保护字符串共用一行时，分隔线和间距要小。

对于必须在 `2x2` 卡片中显示时间 + 标题 + meta 的紧凑提醒/事件行：

- 事件块水平 padding 保持在约 `4-6vp`。
- Row 的 `itemMargin` 保持在约 `3-4vp`。
- 使用细分隔线（`strokeWidth: 1`），或在造成拥挤时省略。
- 当标题和 meta 也必须显示时，`HH:MM` 时间保持约 `18fp`、`48-50vp` 宽，而不是大号 `22fp`/`56vp` 块。
- 同一个紧凑块内共享空间时，短中文标题约 `12fp`，meta 约 `9fp`。

对于横版 `2x4`，假设 root padding 后父级内容宽度通常约 `296vp`。优先使用 2 或 3 个 pane 的主 `Row`，并明确预算每个 pane。不要把额外宽度浪费在过大的装饰间距上；用它保证受保护值完整。

## 按尺寸的交互规则

`2x2`：

- 通常一个主可点击区域或一个全宽动作 pill。
- 避免超过一个 CTA。

`2x4`：

- 一个主动作加一个可选被动状态是可以接受的。
- 如果有多个动作，它们必须视觉上从属、由用户明确要求，并且能放进横向右侧栏或底部条而不被裁剪。

## DataModel 形状

按含义分组数据，而不是按布局分组：

```json
{
  "primary": {"label": "Status", "valueLabel": "31 deg"},
  "context": {"title": "Next meeting", "meta": "09:30 A14"},
  "action": {"label": "Open", "targetId": "demo-001"}
}
```

规则：

- 不要用 `topText1` 这种只反映视觉区域的数据键。
- 当格式化或本地化重要时，预先计算展示字符串。
- 从 DataModel 绑定动作 ID，不要在事件处理器中硬编码业务 ID。

## 生成前自检

写 JSON 前：

- [ ] 我是否明确选择了 `2x2` 或 `2x4`？
- [ ] 我是否识别了语义角色？
- [ ] 结构是否能在尺寸预算内无滚动容纳？
- [ ] 是否只有一个主导焦点？
- [ ] 受保护值是否不会被窄宽度换行破坏？
- [ ] 我是在按规则构造，而不是改造模板吗？
