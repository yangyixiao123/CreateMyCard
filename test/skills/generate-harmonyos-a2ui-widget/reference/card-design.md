# 卡片设计

## 范围

本文档只覆盖紧凑 HarmonyOS GenUI 桌面卡片模式，不覆盖完整页面。

## 卡片边界

- 桌面卡片是一眼可读的摘要容器，不是迷你页面。
- 支持的物理目标是 `2x2`（`160 x 160vp`）和横版 `2x4`（`320 x 160vp`）。
- 默认 root 是一个视觉 shell：带圆角、裁剪、背景/渐变/图片和可选阴影的 `Column` 或 `Stack`。
- HarmonyOS extended catalog 不要求用 `Card` 组件作为桌面卡片 shell；除非真实 `Card` 组件特别有用，否则使用带样式的 `Column` 或 `Stack`。
- 生成由规则驱动。不要选择历史 DSL 模板作为起始结构。

## 内容预算

`2x2` 默认预算：

- 一个主要结论或视觉焦点。
- 一个关键辅助上下文组。
- 一个主动作或状态动作。
- 通常 2 到 3 个主区域。

`2x4` 默认预算：

- 一个主要结论或视觉焦点。
- 两到三个辅助事实，或一个紧凑详情簇。
- 一个主动作或被动状态。
- 通常 2 到 4 个主区域，在同一个 `160vp` 高度内按并排 pane 排列。

默认避免：

- 长列表。
- 长表格。
- 长时间线。
- 多段解释。
- 多个大型并列区块。
- 完整文章/详情页结构。

## 信息摘要卡规则

当一句话请求是实体/概念/地点/事件摘要时：

- 使用 `title -> 1 to 3 attributes -> one short summary -> optional action`。
- 正文必须保持 <= 2 行短文本。
- 默认不要使用动态列表模式。
- 不要使用全宽 16:9 hero 图；这是页面模式。
- 如果需要更多详情，明确建议页面，而不是大卡片。

## 卡片升级规则

把以下情况视为超出卡片范围：

- `2x2` 需要超过 3 个主区域。
- `2x4` 需要超过 4 个主区域。
- 主要信息需要滚动。
- 卡片接近全屏面板。
- 用户需要复杂表单/表格/页面导航。

正确响应：

1. 收敛成摘要卡。
2. 如果无法收敛，使用模式 3 并建议页面范围。
3. 不要悄悄交付页面尺寸卡片。

## 受保护内容可读性

受保护内容包括：

- CTA 标签。
- 时间和日期，例如 `14:00-15:30`。
- 星期标签和用户要求的日期部分。
- 温度和单位，例如 `31 deg`。
- 电量/进度值，例如 `47%`。
- 状态词和短中文短语。
- 用户要求显示的主事件/物品标题。
- 事件倒计时数字和单位。
- 价格、评分和紧凑数量。

规则：

- 除非宽度明显足够，否则不要把受保护内容放入狭窄固定宽列。
- 如果横向空间紧张，先压缩次要描述。
- 弹性文本列使用 `layoutWeight`，只有图标、圆形、图片和视觉仪表使用固定尺寸。
- 对高风险受保护文本使用 `maxLines: 1` 和 `minFontSize`。
- 不要用 `textOverflow: "ellipsis"` 或 `clip` 隐藏受保护内容。改为重新平衡间距、字号、行结构或卡片尺寸。
- 只有在受保护内容已有足够宽度后，才对非必要次要文本使用省略。
- 不要让短短语逐字换行，也不要让单位/标点单独遗留。

## 横向布局策略

- 只有存在真实左右关系时才使用 `Row`：图标 + 标签、指标 + 上下文、图片 + 叠层、动作 + 状态。
- 默认直接 `Row` 子节点 <= 3。
- 先分配角色再定尺寸：
  - 受保护：metric、CTA、icon、status、percentage、time
  - 可压缩：description、subtitle、location、secondary label
- 如果一行有很多小事实，拆成两行或使用 `Column`。
- 如果有 2 个以上标签和一个 CTA，把标签与 CTA 放进 `Column` 的不同行。
- 写拥挤 Row 前，预算其内部宽度：父级宽度减去水平 padding、item margins、分隔线/图标和固定列。如果剩余宽度无法完整显示受保护文本，生成 DSL 前先简化该行。

## 视觉 Shell 指导

- 只保留一个卡片 shell。避免嵌套完整视觉 shell。
- 内部分组使用半透明块、边框、`Divider` 或间距，不要制造多个互相竞争的小卡片。
- 外层圆角应 >= 内层圆角。
- 当图片/渐变可能溢出时，对圆角 root 容器启用裁剪。
- 阴影主要属于 root 或一个聚焦浮动动作，不属于每个小块。

## 缺失数据行为

从稀疏用户输入生成时：

- 仅在自包含演示需要时，推断中性但相关的展示值。
- 按语义分组数据：`weather`、`meeting`、`action`、`asset`、`progress` 等。
- 不要留下示例或历史输出中的样例值。
- 如果缺少真实图片/资源数据，省略 `Image`，使用非媒体视觉技法。

## 图片策略

- `Image.src` 只使用用户提供的真实 URL 或本地资源路径。
- 图片背景卡使用 `Stack` root，`Image` 作为第一个 child，内容叠层作为第二个 child。
- 背景/hero 填充使用 `styles.objectFit: "cover"`。
- 如果图片不能增加场景特定信息，避免使用图片。

## 表现力层级

在 HarmonyOS GenUI extended catalog 内：

- 有真实视觉素材时使用 `Image`。
- 进度/环形/胶囊指示使用 `Progress`。
- 轻量结构使用 `Divider`。
- 层叠光晕/图片/进度组合使用 `Stack`。
- 重复紧凑事实谨慎使用 `Grid`/`List`。
- 当最稳定时，使用普通 `Text` 承载标签、数值和类似图标的字形。

`2x2` 卡片最多一个强表现装置占主导。`2x4` 卡片可以有一个主导装置和一个次级装置。
